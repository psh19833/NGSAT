## Part A. 즉시 수정 (Critical, 실거래 중단 후 진행)

### A-1. KIS API Rate Limit 대응 — refresh_prices() 주기 조정 (Critical, 1시간)
- **관점**: 백엔드
- **현황**: `refresh_prices()`가 10초마다 40종목 × KIS API 호출 = 분당 240회. KIS Research API 제한(분당 20~60회)을 4~12배 초과
- **영향**: 실운영 환경에서 API 응답 실패 → 데이터 미갱신 → 잘못된 매매 결정
- **처리**:
  - (a) `refresh_prices()` 호출 주기를 10초 → **300초(5분)**로 변경
  - (b) 부분 갱신: 상위 5개 후보종목만 매 사이클(10초), 전체 40종목은 5분 주기로 분리
  - (c) KIS WebSocket 실시간 시세 도입 검토 (별도 P2)

```python
# main.py trading_loop() — 수정안
CYCLE_COUNT = 0
while True:
    CYCLE_COUNT += 1
    # 전체 갱신: 30사이클(5분)마다
    if CYCLE_COUNT % 30 == 0:
        universe, index_prices = await data_provider.refresh_prices()
    # 부분 갱신: 매 사이클 상위 후보만 (별도 메서드)
    await data_provider.refresh_top_candidates(orchestrator.top_candidates)
```

- **리스크**: LOW — 주기 변경만으로 기능 변화 없음. 5분 갱신으로 장중 시세 반영 지연은 있으나, 현재 10초 갱신도 실제로는 작동 불가능했으므로 차선

### A-2. auto_tune() — Scaler 일관성 복구 (Critical, 30분)
- **관점**: 백엔드 + 트레이더
- **현황**: `auto_tune()`에서 `self._scaler = None` 설정. 일반 `train()`은 스케일링하지만 튜닝된 모델은 스케일링 없이 추론 → `predict_proba()` 확률 분포 왜곡
- **영향**: 실거래 중 `auto_tune()` 호출 시 갑자기 확률값 의미가 달라져 과매수/과매도 발생
- **처리**: `auto_tune()`에서도 스케일링 파이프라인 유지

```python
# ml/training/trainer.py auto_tune() — 수정안
best_model.fit(X, y)
self._model = best_model
# Scaler 복원: train()과 동일한 파이프라인 유지
from sklearn.preprocessing import StandardScaler
self._scaler = StandardScaler()
self._scaler.fit(X)
self._is_trained = True
```

- **리스크**: LOW — 기존 scaler=None 우회 로직 제거. 새로 튜닝된 모델이 이전과 다른 확률 분포를 출력할 수 있으나, 올바른 방향

### A-3. .env 파일 런타임 변조 중단 — 설정 저장소 분리 (Critical, 2시간)
- **관점**: 백엔드
- **현황**: `api.py`의 `_update_env_from_config()`가 `.env` 파일을 직접 읽고 쓰기(r+). `f.seek(0)+f.truncate()` 사이 크래시 시 파일 손상 위험
- **영향**: 설정 저장 중 서버 크래시 → `.env` 파일이 비어 있거나 잘림 → KIS 인증 정보 상실 → 시스템 기동 불가
- **처리**:
  - (a) **단기(선): `_update_env_from_config()` 제거** — 대시보드 설정 변경 시 `.env`에 쓰지 않고 `orchestrator.config`만 업데이트. 변경사항은 재시작 시 `.env`에서 다시 로드
  - (b) **중기(후): DB 기반 ConfigService 구축** — `system_config` 테이블 + 인메모리 캐시

- **리스크**: MEDIUM (단기) — 대시보드에서 설정 변경 후 재시작 시 변경사항이 사라짐. 사용자에게 "재시작 후 유지" 여부 명시 필요

### A-4. auto_retrain() _last_auc 초기화 (High, 5분)
- **관점**: 트레이더
- **현황**: `PriceRiseModel.__init__()`에 `self._last_auc` 미초기화. 첫 `auto_retrain()` 호출 시 `getattr(self, '_last_auc', 0)`이 항상 0 → 기존 모델 유무와 관계없이 신규 모델로 무조건 교체
- **영향**: ML 모델이 매 재학습 시점마다 교체되어 성능 회귀 가능성
- **처리**: `__init__`에 `self._last_auc = 0.0` 추가

```python
# ml/training/trainer.py — 수정안
self._is_trained = False
self._last_auc = 0.0  # 추가
self._scaler = None
```

- **리스크**: ZERO — 초기화값 추가만으로 기능 변화 없음

### A-5. 5초 폴링 → 개별 컴포넌트 로딩 + 에러 격리 (High, 2시간)
- **관점**: 디자인
- **현황**: `App.jsx`에서 `refreshAll()`이 5개 API를 `Promise.all`로 동시 호출. `setLoading(true)` 전역 설정 → 하나만 실패해도 전체 갱신 실패
- **영향**: API 1개 실패 시 모든 데이터가 빈 상태로 5초간 표시 → 화면 깜빡임
- **처리**:
  - (a) 전역 `setLoading(true)` 제거 → 각 컴포넌트가 독립적 로딩 상태 관리
  - (b) `Promise.all` → 개별 try/catch + 부분 갱신
  - (c) 실패 시 stale data 유지하고 에러 표시만

```javascript
// App.jsx — 수정안
async function refreshAll() {
  const results = await Promise.allSettled([
    fetchStatus().then(setStatus).catch(e => ({ error: e })),
    fetchTrades().then(setTrades).catch(e => ({ error: e })),
    // ...
  ]);
  // 실패한 항목만 toast 알림, 데이터는 유지
}
```

- **리스크**: LOW — 렌더링 로직만 변경. 데이터 흐름에 변화 없음

### B-1. 실거래 미체결 주문 재시도 로직 (High, 1일)
