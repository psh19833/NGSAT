### 1.2 3단계 파이프라인 완성도

```
① 레짐 (Regime) — 완성도 90%
  - KOSPI 일봉 기반 BULL/NEUTRAL/BEAR 0~100점
  - 장중 KOSPI 등락률 ±5점 보정 (TR-16)
  - ADX 추세강도 가중치 추가
  - 5개 가중치: MA 30% + RSI 20% + Bollinger 20%
    + 변화율 15% + 거래량 10%

② 스크리너 (Screener) — 완성도 85%
  - 5개 패턴: 돌파/눌림/반등/볼린저/골든크로스
  - 레짐별 최대 후보: BULL 8 / NEUTRAL 5 / BEAR 2
  - 분봉 스크리너 (minute_screener) 추가
  - KOSPI/KOSDAQ 보너스 가중치

③ ML (Machine Learning) — 완성도 75%
  - 27개 피처 (일봉 20 + 외국인/기관 4 + 재무 3)
  - 5개 모델: RF/GB/XGBoost/LightGBM/LR
  - 자동 재학습 (AUC 개선시에만 교체)
  - Optuna 하이퍼파라미터 최적화
  - 분봉 ML: 25개 피처 (학습 대기중, 데이터 축적중)
```

## 3. 신규 개발 항목 (우선순위별)

### 3.1 우선순위 1 — 즉시 효과, 리스크 낮음 (1~2주)

#### P1-1: 트레일링 스탑 (Trailing Stop Loss)

- **설명**: 수익이 나면 손절선을 따라 올리는 기능
- **비유**: 에어컨 온도를 20도로 맞추면 22도 올라갈 때 자동으로
  20도로 내려주는 것처럼, 주가가 오르면 손절선도 같이 올림
- **효과**: 스윙 모드에서 "수익을 지키면서 추가 상승을 노릴 수 있음"
- **구현 위치**: `live/risk.py` (트레일링 스탑 추적),
  `live/orchestrator.py` (포지션 청산 판단),
  `core/types.py` (Position에 trailing_stop 필드 추가)
- **구현 방식**:
  - ATR 기반: `trailing_stop = current_price - ATR * multiplier`
  - 또는 고정 % 기반: `trailing_stop = max_buy_price * (1 - trail_pct)`
  - 주가 상승시 trailing_stop 은 올라가만 하고, 하락시에는
    올라간 위치에서 멈춤 (ratchet 동작)
- **StrategyConfig 필드**:
  - `trailing_stop_enabled`: bool (기본 False → 안전)
  - `trailing_stop_atr_multiplier`: float (기본 2.0)
  - `trailing_stop_activate_pct`: float (수익 +1%부터 활성화)
- **리스크**: LOW (기존 손절 로직에 additive)
- **의존성**: 없음
- **테스트**:
  - 순수 함수 단위 테스트 (가격 상승→손절선 상승, 하락→유지)
  - 오케스트레이터 통합 테스트 1건 (트레일링 스탑 트리거)
- **예상 공수**: 3~4일

#### P1-2: 부분 청산 (Partial Take-Profit)

- **설명**: 수익 목표 도달시 일부만 매도, 나머지는 추세 추종
- **비유**: 식당에서 음식이 맛있으면 절반 먹고 나머지 포장하는 것
- **효과**: "확정 수익 확보 + 추가 수익 기회 동시 잡기"
- **구현 위치**: `live/orchestrator.py` (청산 판단),
  `live/executor.py` (부분 매도 실행),
  `live/risk.py` (목표 수익률 관리)
- **구현 방식**:
  - 1차 익절: 수익 +3% → 50% 매도
  - 2차 익절: 수익 +6% → 30% 매도
  - 잔여 20%: 트레일링 스탑으로 수익 추적
  - StrategyConfig로 비율/목표 조정 가능
- **StrategyConfig 필드**:
  - `partial_take_profit_enabled`: bool (기본 False)
  - `partial_tp1_pct`: float (1차 익절 수익률, 기본 3.0)
  - `partial_tp1_ratio`: float (1차 매도 비율, 기본 0.5)
  - `partial_tp2_pct`: float (2차 익절 수익률, 기본 6.0)
  - `partial_tp2_ratio`: float (2차 매도 비율, 기본 0.3)
- **리스크**: MEDIUM (포지션 관리 로직 변경, 테스트 필요)
- **의존성**: P1-1 트레일링 스탑과 시너지 (잔여 20% 관리)
- **테스트**:
  - 1차/2차 익절 트리거 단위 테스트
  - 부분 매도 수량 계산 테스트
  - 잔여 포지션 트레일링 스탑 통합 테스트
- **예상 공수**: 4~5일

#### P1-3: 외국인/기관 피처 API 연동 (7개 피처 활성화)

- **설명**: 현재 0값인 7개 ML 피처에 실데이터 주입
- **비유**: 요리에 양념이 빠져있던 것을 넣는 것
- **효과**: "ML 예측 정확도 75% → 78~80% 예상" (핵심 피처)
- **구현 위치**:
  - `data/adapters/kis/endpoints.py` (외국인/기관 매수동향, 재무제표)
  - `data/adapters/kis/mapper.py` (응답 파싱)
  - `data/adapters/kis/adapter.py` (API 호출 메서드)
  - `ml/features/builder.py` (피처 계산에 실데이터 반영)
- **구현 방식**:
  - KIS API 엔드포인트 추가 (additive 패턴):
    1. 외국인/기관 일별 매수동향: TR_ID 확인 필요
    2. 재무제표 (PER/PBR/EPS): TR_ID 확인 필요
  - mapper.py에 순수 파싱 함수 추가
  - builder.py에서 API 데이터 → 피처 계산
  - 기존 0값 fallback 유지 (API 실패시 안전)
- **KIS API TR_ID** (사전 확인 필요):
  - 외국인/기관 매수동향: KIS open-trading-api 예제 코드에서 확인
  - 재무제표: KIS open-trading-api 예제 코드에서 확인
- **리스크**: LOW (additive, 기존 로직 변경 없음)
- **의존성**: KIS API endpoints.py 추가 (스킬의 KIS 확장 패턴 사용)
- **테스트**:
  - endpoints 등록 테스트
  - mapper 파싱 테스트 (순수 함수)
  - builder 피처 계산 테스트 (실데이터 vs fallback)
- **예상 공수**: 3~4일

#### P1-4: test_risk.py 3개 에러 수정

- **설명**: 일일손실 한도 테스트 3개가 실패중
- **테스트 목록**:
  - `test_safe_daily_loss`
  - `test_daily_loss_limit_reached`
  - `test_halt_can_be_reset`
- **효과**: "테스트 305→308 전체 통과로 신뢰성 회복"
- **구현 위치**: `tests/test_live/test_risk.py`
- **리스크**: LOW (테스트 수정)
- **의존성**: 없음
- **예상 공수**: 1~2시간

### 3.2 우선순위 2 — 중기 효과, 리스크 중간 (2~4주)

#### P2-1: 드로우다운 관리 (Max Drawdown Circuit Breaker)

- **설명**: 누적 손실이 한도 초과시 자동 매매 중지
- **비유**: 자동차 계기판에서 연료가 빨간색이 되면 정비소 가듯,
  계좌가 일정 수준 이상 깎이면 잠시 쉼
- **효과**: "연속 손실일으로 인한 큰 손실 방지"
- **구현 위치**: `live/risk.py` (드로우다운 추적),
  `live/orchestrator.py` (매매 중지 판단),
  `core/types.py` (DrawdownState 추가)
- **구현 방식**:
  - 7일 이동 손실 추적 (rolling drawdown)
  - 최대 드로우다운 한도 (예: -10%) 초과시 controller.pause()
  - 텔레그램 알림: "최대 드로우다운 한도 도달 — 매매 일시정지"
  - 수동 해제 (대시보드 버튼) 또는 다음 날 자동 리셋
- **StrategyConfig 필드**:
  - `max_drawdown_pct`: float (기본 10.0)
  - `drawdown_window_days`: int (기본 7)
  - `drawdown_auto_reset`: bool (기본 True — 매일 장 시작시 리셋)
- **리스크**: MEDIUM (매매 중지 로직 → 대표님 확인 필요)
- **의존성**: 없음
- **테스트**:
  - 드로우다운 계산 단위 테스트
  - 한도 도달시 일시정지 테스트
  - 자동/수동 리셋 테스트
- **예상 공수**: 3~4일

#### P2-2: 업종 분산 관리 (Sector Diversification)

- **설명**: 같은 업종에 과도하게 투자하지 않도록 제한
- **비유**: 계란을 한 바구니에 담지 않는 것
- **효과**: "특정 업종 하락시 전체 포지션 손실 완화"
- **구현 위치**: `live/risk.py` (업종 비중 검사),
  `live/orchestrator.py` (매수 전 검사),
  `data/real_data_provider.py` (종목→업종 매핑)
- **구현 방식**:
  - 종목→업종 매핑 테이블 (하드코딩 또는 KIS API)
  - 업종별 최대 비중 (예: 전체 포지션의 30%)
  - 동일 업종 최대 종목 수 (예: 3종목)
  - 매수 전 검사: "이 업종에 이미 3종목 있으면 진입 생략"
- **StrategyConfig 필드**:
  - `sector_max_pct`: float (기본 0.30)
  - `sector_max_stocks`: int (기본 3)
- **리스크**: MEDIUM (데이터 구조 변경, 업종 매핑 필요)
- **의존성**: 종목 메타데이터 (업종 정보)
- **테스트**:
  - 업종 비중 계산 테스트
  - 한도 초과시 진입 차단 테스트
- **예상 공수**: 3~5일

#### P2-3: 앙상블 ML 모델 (Soft Voting)

- **설명**: 3개 모델(XGBoost+LightGBM+RF)의 예측을 가중 평균
- **비유**: 의사 3명의 의견을 모아서 진단하는 것
- **효과**: "단일 모델 대비 안정적 예측, 시장 변화에 강함"
- **구현 위치**: `ml/inference.py` (앙상블 래퍼),
  `ml/training/trainer.py` (3개 모델 동시 학습)
- **구현 방식**:
  - 각 모델별 predict_proba 개별 계산
  - 가중 평균: `ensemble_prob = w1*p1 + w2*p2 + w3*p3`
  - 가중치는 각 모델의 AUC 기반 자동 설정
  - 모델 저장: 3개 pkl 파일 + 가중치 메타데이터
- **StrategyConfig 필드**:
  - `ml_ensemble_enabled`: bool (기본 False)
  - `ml_ensemble_weights`: str ("auto" 또는 "0.4,0.3,0.3")
- **리스크**: MEDIUM (추론 시간 증가, 메모리 사용 증가)
- **의존성**: P1-3 (피처 활성화 후 성능 비교 필요)
- **테스트**:
  - 앙상블 가중 평균 계산 테스트
  - 단일 모델 fallback 테스트
- **예상 공수**: 3~4일

#### P2-4: 백테스트 유니버스 확장 + 거래비용 정밀화

- **설명**: KOSPI 30 → KOSPI+KOSDAQ 50~100종목, 정확한 세금/수수료
- **비유**: 더 넓은 바다에서 낚시하되, 낚시비용을 정확히 계산
- **효과**: "더 많은 기회 포착 + 현실적인 백테스트 결과"
- **구현 위치**: `data/real_data_provider.py` (유니버스 확장),
  `backtest/engine.py` (거래비용),
  `backtest/data_loader.py` (코스닥 데이터)
- **구현 방식**:
  - 코스닥 주요 종목 20~50개 추가
  - 거래비용: 매매세금 0.23% (선취수수료 0.015%)
  - 백테스트 결과에 거래비용 항목 추가
- **리스크**: MEDIUM (API 호출 증가 → Rate Limit 관리)
- **의존성**: Rate Limit 개선 후
- **테스트**:
  - 거래비용 계산 테스트
  - 확장 유니버스 백테스트 smoke test
- **예상 공수**: 3~5일

### 3.3 우선순위 3 — 장기 효과, 리스크 높음 (1~3개월)

#### P3-1: 분봉 ML 모델 학습 및 단타 모드 활성화

- **설명**: 축적된 분봉 데이터로 단타 ML 학습 → SHORT_TERM 모드 실사용
- **비유**: 단타 트레이더를 새로 채용해서 교육시키는 것
- **효과**: "당일 청산 단타 매매 활성화, 수익 기회 확대"
- **전제 조건**: ~60 영업일 분봉 데이터 (현재 약 5일, 약 11주 후 가능)
- **구현 위치**: `ml/training/trainer.py`, `ml/inference.py`,
  `live/orchestrator.py`
- **리스크**: HIGH (매매 흐름 변경, 실거래 진입)
- **의존성**: 분봉 데이터 축적 (cron 실행 지속 필요)
- **예상 공수**: 데이터 축적 11주 + 학습/검증 1~2주

#### P3-2: 딥러닝 시계열 모델 (LSTM/Transformer)

- **설명**: 시계열 패턴을 학습하는 딥러닝 모델 추가
- **비유**: 트리 기반 모델은 사진을 보고 판단하지만, LSTM은
  영상(흐름)을 보고 판단
- **효과**: "차트 패턴의 시간적 흐름 학습 → 예측 정확도 향상"
- **구현 위치**: `ml/` 하위 새 모듈 (deep/ 디렉토리)
- **리스크**: HIGH (학습 시간, GPU 필요 가능성, 복잡도 증가)
- **의존성**: 충분한 데이터 + P1-3 피처 활성화
- **예상 공수**: 2~4주 (데이터 준비 후)

#### P3-3: 강화학습 (Deep Reinforcement Learning)

- **설명**: 시장 상황에 따라 최적 행동을 스스로 학습
- **비유**: 체스 AI가 수백만 판을 두며 실력을 키우듯, 매매도
  시뮬레이션으로 최적화
- **효과**: "시장 변화에 자동 적응, 정적 규칙 대비 유연한 대응"
- **구현 위치**: 새 모듈 (rl/ 디렉토리, 백테스트 환경 기반)
- **리스크**: HIGH (복잡도 매우 높음, 백테스트 환경 필수)
- **의존성**: P3-1 분봉 ML + 정교한 백테스트 엔진
- **예상 공수**: 1~3개월

#### P3-4: 휴일 캘린더 자동화

- **설명**: KIS 휴일 API 또는 한국거래소 휴일 정보 연동
- **효과**: "공휴일/임시휴장시 시스템 오작동 방지"
- **구현 위치**: `data/adapters/kis/endpoints.py` + `adapter.py`
- **리스크**: LOW (additive)
- **의존성**: KIS 휴일 API TR_ID 확인 필요
- **예상 공수**: 1~2일

#### P3-5: PostgreSQL 마이그레이션

- **설명**: SQLite → PostgreSQL 영속화
- **효과**: "동시성 안정성, 대용량 데이터 관리, 프로덕션 수준"
- **구현 위치**: `data/db.py` (엔진 전환), `core/models.py`
- **리스크**: MEDIUM (스키마 마이그레이션)
- **의존성**: 스키마 이미 준비됨 (docs에 PostgreSQL 설정 가이드 있음)
- **예상 공수**: 2~3일

## 4. 추천 진행 순서

```
주말 1차 (P1-4 + P1-1):
  ① P1-4 테스트 에러 수정 (1시간)
  ② P1-1 트레일링 스탑 구현 (3~4일)

주말 2차 (P1-3 + P1-2):
  ③ P1-3 외국인/기관 피처 API 연동 (3~4일)
  ④ P1-2 부분 청산 구현 (4~5일)

주말 3차 (P2-1 + P2-3):
  ⑤ P2-1 드로우다운 관리 (3~4일)
  ⑥ P2-3 앙상블 ML 모델 (3~4일)

이후 (P2-2 + P2-4):
  ⑦ P2-2 업종 분산 관리 (3~5일)
  ⑧ P2-4 백테스트 유니버스 확장 (3~5일)

장기 (데이터 축적 후):
  ⑨ P3-1 분봉 ML 학습 + 단타 모드 (~11주 후)
  ⑩ P3-2~3 딥러닝/강화학습 (선택적)
  ⑪ P3-4 휴일 캘린더 (틈날 때)
  ⑫ P3-5 PostgreSQL 마이그레이션 (필요시)
```

진행 흐름의 원칙:
- 수익 관리 강화 (트레일링 스탑 → 부분 청산)
- ML 성능 향상 (피처 활성화 → 앙상블)
- 리스크 관리 강화 (드로우다운 → 업종 분산)
- 각 단계는 독립적, 리스크가 낮은 것부터
- 모든 항목은 StrategyConfig로 설정 가능 (대표님이 조정)
- 기본값은 False (안전) → 테스트 후 활성화
