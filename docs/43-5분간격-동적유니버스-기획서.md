# 5분 간격 동적 유니버스 교체 — 기획서

**버전:** 1.0
**작성일:** 2026-07-02
**상태:** 기획

---

## 1. 개요

장 시작 후 5분 간격으로 유니버스 40종목 중 점수 하위 50%(20종목)를 새로운 종목으로 교체하여, **항상 시장에서 가장 활발하고 스코어가 높은 종목**을 매매 대상으로 유지한다.

---

## 2. 장중 타임라인

```
09:00 ─── 장 시작
  │  KIS 거래량순위 API → 상위 100종목
  │  → 일봉 스크리너로 100종목 평가 → 점수順 40종목 선정
  │  → 100종목 일봉 데이터 로드 (100 REST)
  │  → WebSocket 40종목 구독 + 매매 시작
  │
09:05 ─── 1차 교체
  │  거래량순위 API (1 REST) → 상위 100종목
  │  → 현재 40종목 평가 → 하위 20종목 탈락
  │  → 상위 100 - 현재 40 = 60 후보 → 스크리너 평가 → 상위 20종목 편입
  │  → WebSocket 구독 교체 (-20 +20)
  │
09:10 ─── 2차 교체 (동일 방식)
       ...
15:25 ─── 마지막 교체
15:30 ─── 장 마감
```

## 3. 리서치: 새 종목 편입 방법

### 3.1 비교 분석

| 방법 | REST 호출 | 장점 | 단점 |
|:----|:--------:|:-----|:-----|
| **A: 거래량순위 API 5분 재호출** | 1회/5분 (총 78회) | 항상 최신 거래량 반영 | 신규 종목 일봉 로드 필요 |
| **B: 고정 예비 리스트(100종목)** | 0회/장중 | REST 부하 없음 | 예비 리스트가 고정됨 |
| **C: 하이브리드 (추천)** | 1회/5분 (총 ~88회) | 최신 거래량 + 사전로드 활용 | 약간 복잡 |

### 3.2 추천: 하이브리드 방식 (C)

**① 09:00 — 초기 100종목 사전 로드**

```
거래량순위 API → 상위 100종목
  ↓
일봉 스크리너 100종목 평가
  ↓
TOP 40 = 활성 유니버스 + 나머지 60 = 예비 리스트
  ↓
100종목 × 360일 일봉 데이터 로드 (100 REST, 최초 1회)
  ↓
WebSocket 40종목 구독
```

**② 5분마다 — 교체**

```
거래량순위 API 1회 호출 (1 REST)
  ↓
현재 40종목 스크리너 평가 → 하위 20종목 선정 (탈락)
  ↓
예비 60종목 스크리너 평가 → 상위 20종목 선정 (편입)
  ↓
예비 리스트 갱신: volume-rank 상위 100 - 현재 40 = 60종목
  ↓
WebSocket 구독 교체: unsubscribe 20 + subscribe 20
```

**③ 신규 종목 등장 시 (예비 리스트에 없는 종목)**

```
5분마다 받은 거래량순위 상위 100종목 중
  → 예비 리스트에 없는 종목 발견
  → 일봉 데이터 1회 로드 (1 REST)
  → 예비 리스트에 추가
```

**REST 호출 계산:**
```
초기:   100회 (일봉 로드)
5분마다: 1회 (거래량순위) × 78회 = 78회
신규:    ~10회 (예비 리스트 없는 종목)

총:     ~188회/일 (초당 0.002회, Rate Limit 여유 충분)
```

## 8. 보유 포지션 보호

```python
def _select_for_removal(active_40, held_codes) -> list:
    """하위 20종목 선정, 단 보유 포지션은 보호."""
    sorted_by_score = sorted(active_40, key=lambda x: x.score)
    candidates = [s for s in sorted_by_score if s.code not in held_codes]
    return candidates[:20]  # 최대 20개만 교체
```

보유 포지션이 20개 이상이면 교체 자체를 건너뜁니다.
