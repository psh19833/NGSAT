## 1. 개요

장 시작 후 5분 간격으로 유니버스 40종목 중 점수 하위 50%(20종목)를 새로운 종목으로 교체.
단, **거래량 단독이 아닌 4개 축을 결합한 다중 요인 평가**로 종목을 선정한다.

---

## 2. 종목 선정: 4개 축 결합 점수

### 2.1 평가 축

| 축 | 출처 | REST | 측정 대상 | 비중 |
|:--|:-----|:---:|:---------|:---:|
| **① 거래량순위** | `volume_rank` API | 1회 | 시장 활동성 (얼마나 거래되는가) | **25%** |
| **② 체결강도순위** | `volume_power` API | 1회 | **매수압력** (누가 사고 있는가) | **20%** |
| **③ 등락률순위** | `fluctuation` API | 1회 | 단기 모멘텀 (방향성) | **25%** |
| **④ 스크리너 기술점수** | 자체 계산 (기존 일봉) | **0회** | RSI/MACD/MA/ADX/패턴 | **30%** |

### 2.2 왜 거래량만 안 되는가

```
거래량만 많은 종목 = 
  기관 대량 매매 (괜찮음)
  or 패닉 셀링 (위험)
  or 뉴스 재료 소진 (늦음)

체결강도 + 등락률 + 스크리너 추가 시 =
  "활발하게 거래되면서 + 매수세가 강하고 + 상승 추세이며 + 기술적으로 건전한" 종목
```

### 2.3 점수 계산 방식

```
종목 선정 점수 = 
    normalize(거래량순위) × 0.25
  + normalize(체결강도순위) × 0.20
  + normalize(등락률순위) × 0.25
  + normalize(스크리너점수) × 0.30
```

`normalize(rank)` = `max(0, 100 - rank)`로 1등 100점, 100등 0점으로 정규화.

## 3. 장중 타임라인

```
09:00 ─── 장 시작
  │  multi_rank() 호출 (3 REST):
  │    ┌ volume_rank → 거래량순위 100
  │    ├ volume_power → 체결강도순위 100
  │    └ fluctuation → 등락률순위 100
  │  → 4개 축 점수 합산 → 100종목 순위
  │  → TOP 40 = 활성 유니버스 (+ 나머지 60 = 예비)
  │  → 100종목 일봉 데이터 로드 (100 REST, 최초 1회)
  │  → WebSocket 40종목 구독
  │
09:05 ─── 1차 교체 (3 REST):
  │  multi_rank() 재호출
  │  → 현재 40종목 재평가 → 하위 20종목 제외 (보유포지션 보호)
  │  → 예비 60 + 신규 100 비교 → 상위 20종목 편입
  │  → WebSocket -20 +20 교체
  │
09:10 ─── 2차 교체
... 5분 간격 반복 ...
15:25 ─── 마지막 교체
15:30 ─── 장 마감
```

### 4.2 호출량 계산

| 구분 | 09:00 | 5분마다 | 일일 합계 |
|:----|:----:|:-------:|:--------:|
| 거래량순위 | 1회 | 1회 | 79회 |
| 체결강도순위 | 1회 | 1회 | 79회 |
| 등락률순위 | 1회 | 1회 | 79회 |
| 스크리너 | 0회 (자체 계산) | 0회 | **0회** |
| 일봉 로드 (초기 100종목) | 100회 | 0회 | 100회 |
| 신규 종목 일봉 | — | ~2회 | ~20회 |
| **합계** | **103회** | **3회** | **~357회/일** |

Rate Limit: 초당 20회 = 시간당 72,000회. **357회는 0.5% 미만으로 여유 충분.**

| 항목 | 리스크 | 사유 | 대책 |
