- 리스크: LOW (additive, 기본값 False)

### 2.2 세부 단계

#### 2단계-A: Position 데이터클래스 확장 (core/types.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| A1 | trailing_stop_price 필드 추가 | `trailing_stop_price: Optional[float] = None` — 현재 트레일링 스탑 가격 |
| A2 | trailing_stop_high_water 필드 추가 | `trailing_stop_high_water: Optional[float] = None` — 포지션 보유 중 최고가 |
| A3 | frozen=True 유지 | dataclass가 frozen이므로, 새 Position 객체를 생성하여 교체하는 패턴 사용 |

리스크: Position은 KIS mapper에서 생성됨. 새 필드는 Optional + 기본값 None이므로
기존 mapper 코드 변경 불필요. 단, orchestrator에서 Position을 업데이트할 때
새 객체를 생성하는 패턴이 필요 (frozen dataclass이므로 `dataclasses.replace` 사용).

#### 2단계-B: StrategyConfig 필드 추가 (core/config.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| B1 | trailing_stop_enabled 필드 | `trailing_stop_enabled: bool = False` |
| B2 | trailing_stop_atr_multiplier | `trailing_stop_atr_multiplier: float = 2.0` |
| B3 | trailing_stop_activate_pct | `trailing_stop_activate_pct: float = 1.0` — 수익 +1%부터 활성화 |
| B4 | .env 변수 추가 | NGSAT_TRAILING_STOP_ENABLED=false 등 |
| B5 | .env.example 업데이트 | 새 변수 설명 추가 |

#### 2단계-C: RiskManager에 트레일링 스탑 로직 추가 (live/risk.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| C1 | update_trailing_stop() 메서드 | Position과 현재가를 받아: (1) high_water 업데이트, (2) ATR 기반 trailing_stop_price 계산, (3) 기존 stop보다만 올림 (ratchet), (4) 새 Position 반환 (frozen replace) |
| C2 | check_trailing_stop() 메서드 | Position의 current_price가 trailing_stop_price 이하면 청산 권고 RiskCheckResult 반환 |
| C3 | 활성화 조건 | profit_loss_pct >= trailing_stop_activate_pct일 때만 활성화 |
| C4 | 비활성시 fallback | trailing_stop_enabled=False이면 기존 고정 손절선 로직 유지 |

리스크: 새 메서드 추가이므로 기존 check_stop_loss/check_daily_loss에 영향 없음.
리스크: ATR 계산에 필요한 가격 데이터가 부족할 때의 fallback 필요 (기존 손절선 유지).

### 4.2 세부 단계

#### 4단계-A: Position 데이터클래스 확장 (core/types.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| A1 | partial_tp_executed 필드 추가 | `partial_tp_executed: bool = False` — 1차 익절 완료 여부 |
| A2 | partial_tp2_executed 필드 추가 | `partial_tp2_executed: bool = False` — 2차 익절 완료 여부 |
| A3 | original_quantity 필드 추가 | `original_quantity: Optional[int] = None` — 최초 매수 수량 (잔량 계산용) |

리스크: Position은 KIS mapper에서 생성됨. 새 필드는 Optional + 기본값이므로
mapper 코드 변경 불필요. orchestrator에서 dataclasses.replace()로 업데이트.

#### 4단계-B: StrategyConfig 필드 추가 (core/config.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| B1 | partial_tp_enabled | `partial_tp_enabled: bool = False` |
| B2 | partial_tp1_pct | `partial_tp1_pct: float = 3.0` — 1차 익절 수익률 |
| B3 | partial_tp1_ratio | `partial_tp1_ratio: float = 0.5` — 1차 매도 비율 (50%) |
| B4 | partial_tp2_pct | `partial_tp2_pct: float = 6.0` — 2차 익절 수익률 |
| B5 | partial_tp2_ratio | `partial_tp2_ratio: float = 0.3` — 2차 매도 비율 (30%) |
| B6 | .env 변수 추가 | NGSAT_PARTIAL_TP_ENABLED=false 등 |
| B7 | CONFIG_FIELD_MAP 업데이트 | dashboard/backend/api.py |
| B8 | presets.json 업데이트 | 8개 프리셋에 partial_tp_* 값 추가 |

리스크: P1-1과 동일한 3곳 동시 업데이트 규칙.

#### 4단계-C: RiskManager에 부분 청산 판단 로직 (live/risk.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| C1 | check_partial_take_profit(position) 메서드 | 현재 수익률 기준: (1) tp1 도달 + 미실행 → 1차 권고, (2) tp2 도달 + 미실행 → 2차 권고, (3) 이미 실행 → NONE |
| C2 | 반환 타입 | `PartialTPResult(should_sell: bool, sell_quantity: int, tp_stage: int, reason: str)` |
| C3 | sell_quantity 계산 | 1차: `int(original_qty * tp1_ratio)`, 2차: `int(original_qty * tp2_ratio)` |
| C4 | 잔량 계산 | 전체 - 1차 - 2차 = 잔여 (트레일링 스탑이 관리) |
| C5 | 비활성시 | enabled=False → PartialTPResult(should_sell=False) |

리스크: 수량 계산시 int 변환에서 0주가 될 수 있음 → 최소 1주 보장 로직.

#### 4단계-D: Orchestrator Step 7에 부분 청산 연동 (live/orchestrator.py)

| 단계 | 작업 | 상세 |
|------|------|------|
| D1 | Step 7 루프에 부분 청산 검사 추가 | 손절/ML청산 검사 전에 부분 익절 검사 |
| D2 | 부분 매도 실행 | `executor.execute_sell(quantity=sell_quantity)` — 전량이 아닌 부분 수량 |
| D3 | Position 업데이트 | `dataclasses.replace(position, quantity=remaining, partial_tp_executed=True)` |
| D4 | Position DB 동기화 | 잔량 변경 후 position_record 업데이트 |
| D5 | DecisionReason | "부분 익절 1차: 수익 +3% 달성, 50% 매도 (X주/Y주)" |
| D6 | 텔레그램 알림 | 부분 익절 발생시 알림 (선택적) |

리스크: orchestrator 로직 변경 범위가 가장 큼. 기존 전량 매도 흐름에
부분 매도 분기를 추가하는 것이므로, 기존 흐름을 깨지 않는지 주의.
리스크: Position이 frozen이므로 replace() 사용 → position 목록 갱신 필요.
리스크: KIS에 부분 매도 후 잔량이 정확히 반영되는지 확인 필요 (KIS 잔고 조회).
