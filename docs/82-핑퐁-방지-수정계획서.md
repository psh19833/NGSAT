## 2. 원인 분석

### 청산 파이프라인 (exit_manager.py _check_single_exit)

```
Step 1: 트레일링 스탑 체크    →  skip (trailing_stop_enabled=false)
Step 2: 부분 익절 체크        →  skip (partial_tp_enabled=false)
Step 3: 손절선 체크           →  0.09% < 1.5%  → skip
Step 4: 분봉 선제 청산        →  ±0.09%는 급락/과열 조건 미달 → skip
Step 5: ML 청산 예측          →  ← *** SELL 반환 → 청산 실행 ***
```

**Step 5가 핑퐁의 직접적 원인.**

### ML 예측의 불안정성

분봉ML 진입(30bars, rise_prob ≥ 60%) 후 다음 사이클(10초):
- 분봉 1~2개(10~20초)의 데이터가 추가됨
- 새로운 분봉 데이터로 예측 확률이 급변
- 확률이 청산 기준(30%) 이하로 떨어지면 즉시 SELL
- 다음 사이클에서 다시 BUY 조건 충족 → 재진입

### Position 객체의 현재 구조 (core/types.py:130)

```python

    name: str
    quantity: int
    buy_price: float
```

**Position에 진입 시각(entry_time)이 없어서** 몇 분 전에 매수했는지 알 수 없음.

## 4. 수정 상세

### 4.1 core/types.py — Position에 entry_time 추가

```python
class Position:
    code: str
    name: str
    quantity: int
    buy_price: float
    current_price: float
    ...
    entry_time: float = 0.0   # time.time() — min_hold 계산용
```

- 기본값 0.0으로 하위호환성 유지
- 변경 시 Position 객체를 생성하는 모든 곳에서 `entry_time`을 설정해야 함

### 4.2 orchestrator.py — 포지션 조회 시 entry_time 매핑

포지션을 broker에서 조회해올 때, 기존 DB PositionRecord에서 매수 시각을 가져와서 매핑.

또는 `TradeRecorder`나 `ExitManager`가 Position이 생성될 때 `entry_time = time.time()`을 설정.

**가장 간단한 방법:** `exit_manager._check_single_exit()` 내부에서 position.entry_time이 0.0이면 `time.time()`으로 초기화. (첫 번째 청산 체크 시각을 진입 시각으로 간주)

### 4.3 exit_manager.py — min_hold 체크 추가

```python
# _check_single_exit() 상단, 청산 조건 검사 전에 추가

# ── 핑퐝 방지: 최소 보유 시간(min_hold) ──
MIN_HOLD_MINUTES = 20
now = time.time()
if position.entry_time == 0.0:
    position.entry_time = now  # 첫 진입 시각 기록
elapsed_min = (now - position.entry_time) / 60
if elapsed_min < MIN_HOLD_MINUTES:
    # 손절선은 예외적으로 허용 (급락 보호)
    loss_pct = abs(min(position.profit_loss_pct, 0))
    base_stop = self._risk.effective_stop_loss_pct or position.stop_loss_pct
    effective_stop = self._risk.regime_adjusted_stop_loss(base_stop)
    if loss_pct < effective_stop:
        # min_hold 내에는 stop_loss 외 청산 금지
        return None
```

## 5. 리스크 평가

| 항목 | 리스크 | 내용 |

| **ML 청산 지연** | 🟡 20분 후에는 정상 작동 | 20분 내 급락은 stop_loss가 방어 |
