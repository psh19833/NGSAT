## M-11: w_sum 정규화 미흡 — `strategy/regime.py`

### 현황

```python
# L140-158 (현재 코드)
adx_w = cfg.regime_weight_adx
w_sum = w_ma + w_rsi + w_bb + w_cr + w_vol + adx_w
if abs(w_sum - 100.0) > 0.01:
    logger.warning(...)
    w_sum = max(w_sum, 1.0)  # division by zero 방지
else:
    w_sum = 100.0
total = (...) / w_sum
```

`w_sum != 100`이면 divisor만 보정하고 개별 가중치는 재조정 안 함.
→ `w_sum=95`일 때 `w_ma=30`이지만 실제 기여도는 `30/95 ≈ 31.6%`로 의도(30%)와 다름.

### 로그 검증 결과

- **설정값:** Config 기본값 + DB 오버라이드 → `MA=30 + RSI=20 + BB=20 + CR=15 + VOL=10 + ADX=5 = 100.0`
- **w_sum 경고 로그:** **0건** (07/06~07/13 전수 검색)
- **실제 운용 영향:** **없음.** 설정이 변경되지 않는 한 절대 발동 안 함.

### 수정안

**A안 (권장): 개별 가중치 비례 재조정**

```python
if abs(w_sum - 100.0) > 0.01:
    logger.warning(f"레짐 가중치 합이 100이 아님: {w_sum:.1f} — 개별 가중치 비례 재조정")
    scale = 100.0 / max(w_sum, 1.0)
    w_ma *= scale
    w_rsi *= scale
    w_bb *= scale
    w_cr *= scale
    w_vol *= scale
    adx_w *= scale
    w_sum = 100.0
total = (...) / w_sum  # 항상 100으로 나눔
```

**B안: 변경 없음 (사유: 실제 발동 이력 0건)**
- 방어코드만 유지하고 기능 변경 없음

**추천: A안.** 3줄 추가로 방어 로직 완전성 확보.

### 리스크
🟢 개별 가중치 재조정은 정규화의 표준 방식. 소수점 오차 이내 영향.

## N3: executor.py 300ms 고정 sleep — `live/executor.py`

### 현황

```python
# L312 (execute_buy) / L408 (execute_sell) 
await asyncio.sleep(0.3)
actual_fill = await self._broker.get_fill_price(order_id)
```

시장가/지정가 주문 체결 후 0.3초 대기 후 체결가 조회. 이 sleep은 KIS API에 체결이 반영될 시간을 주기 위한 것.

### 로그 검증 결과

- **get_fill_price 로그:** **07/10~07/13 모든 호출이 HTTP 404 실패** (20건 전수)
  ```
  WARNING | 체결가 조회 중 오류 (0021640300): KIS HTTP error on inquire_order: HTTP 404
  WARNING | 체결가 조회 중 오류 (0002842600): KIS HTTP error on inquire_order: HTTP 404
  ```
- **실제 fill_price 사용:** `actual_fill=0`으로 fallback → `fill_price`(limit price 또는 시장가 추정가) 사용
- **0.3초 지연:** **매 주문마다 300ms 추가 대기** — 체결가 조회는 항상 실패하므로 순수 낭비

### 수정안

**A안 (권장): sleep 제거 + 조회 자체 조건부 축소**

```python
# P-81: 체결가 조회 — 지정가(price!=None)는 sleep 불필요, 시장가만 0.2초 대기
actual_fill = 0.0
if order_id and adapted_price is None:  # 시장가 주문만
    try:
        await asyncio.sleep(0.2)  # 300→200ms
        actual_fill = await self._broker.get_fill_price(order_id)
    except Exception:
        logger.debug(f"체결가 조회 실패: {code} {order_id}")
```

**변경 요점:**
1. 시장가 주문(`adapted_price is None`)일 때만 체결가 조회 (지정가=limit이므로 fill_price = limit_price)
2. 0.3→0.2초로 단축
3. 실패 시 debug 로그만 (warning flood 방지)

### 리스크
🟢 지정가 주문은 limit가 곧 fill_price이므로 조회 불필요. 시장가만 조회해도 충분.

## N4: backtest market 휴리스틱 — `backtest/engine.py`

### 현황

```python
# L457 (backtest engine only, 1곳)
market=Market.KOSPI if pred.code[:1] in ("0", "1") else Market.KOSDAQ,
```

6자리 종목코드 첫자리가 "0" 또는 "1"이면 KOSPI, 아니면 KOSDAQ으로 추정.
"0" 시작 코드 중 일부는 코스닥 우선주/ETF일 수 있음.

### 로그 검증 결과

- **live 코드 사용:** **0건.** 이 heuristic은 `backtest/engine.py` 1곳만 사용
- **DB 내 "0" 시작 코드:** **0개** (market_data_cache 기준)
- **실제 운용 영향:** **0건.** 백테스트 시에만 영향
- **영향 범위:** "095570"(AJ네트웍스, KOSDAQ) 같은 극소수 코드가 "0"으로 시작

### 수정안

**A안 (권장): StockInfo에서 market 필드 전달**

StockInfo 객체에 이미 market 필드가 있거나 추가 가능한지 확인. `Prediction` 객체에 `market` 정보를 함께 전달.

```python
# backtest/engine.py
# 현재: market=Market.KOSPI if pred.code[:1] in ("0", "1") else Market.KOSDAQ
# 수정: pred.market이 있으면 사용, 없으면 기존 heuristic fallback
market = pred.market if hasattr(pred, 'market') and pred.market else (
    Market.KOSPI if pred.code[:1] in ("0", "1") else Market.KOSDAQ
)
```

**B안: 2자리 휴리스틱으로 개선**

```python
...
```

**추천: A안.** 휴리스틱 자체가 live에 영향 없고 P3 수준이므로, StockInfo 연동이 맞으면 A안으로.

### 리스크
🟢 **live 영향 0.** 백테스트에서 극소수 종목(0시작 코스닥) 오분류 가능하나 실거래와 무관.
