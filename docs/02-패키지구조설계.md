## 2. 패키지 의존성 규칙

### 허용되는 import 방향

```
         ┌──────────────────────────────────┐
         │           core/                  │  ← 모든 패키지가 import 가능
         ├──────────────────────────────────┤
         │           data/                  │  ← core, data import 가능
         ├──────────────────────────────────┤
         │         strategy/                │  ← core, data import 가능
         ├──────────────────────────────────┤
         │            ml/                   │  ← core, data, strategy import 가능
         ├──────────────────────────────────┤
         │     live/        backtest/       │  ← core, data, strategy, ml import 가능
         │                                         BUT live ↔ backtest import 금지
         └──────────────────────────────────┘
```

### 규칙 요약

| 패키지 | import 가능 | import 불가 |
|--------|-------------|-------------|
| core | (없음) | 모든 하위 패키지 |
| data | core | 나머지 전부 |
| strategy | core, data | ml, live, backtest, dashboard, telegram |
| ml | core, data, strategy | live, backtest, dashboard, telegram |
| live | core, data, strategy, ml | backtest, dashboard, telegram |
| backtest | core, data, strategy, ml | live, dashboard, telegram |
| dashboard | core, data, strategy, ml | live, backtest |
| telegram | core, data, strategy, ml, live | backtest, dashboard |

## 3. 핵심 설계 원칙

### 3.1 한 모듈 = 한 책임
- 각 파일/클래스는 하나의 명확한 역할만 수행
- 예: `risk.py`는 리스크 계산만, `executor.py`는 주문 실행만

### 3.2 의존성 역전 (DIP)
- 상위 모듈이 하위 모듈에 직접 의존하지 않음
- 인터페이스(abstraction)를 통해 의존
- 예: `live/executor.py` → `data/adapters/base.py` (인터페이스) → `data/adapters/kis/adapter.py` (구현)

### 3.3 모든 매매 판단에 근거 명시
- 매수/매도/홀드/손절 결정 시 DecisionReason 객체 생성
- DecisionReason은 데이터베이스에 저장되고 알림에 포함됨

### 3.4 설정 외부화
- 모든 설정값은 .env 또는 데이터베이스에서 관리
- 코드에 하드코딩된 매직 넘버 금지
