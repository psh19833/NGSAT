## 1. 분봉ML 예측 불안정 개선 (근본 원인)

### 현황
- 분봉ML AUC 0.674 (랜덤 0.5 대비 간신히 우위)
- 동일 종목에 대해 연속된 사이클(10초 간격)에서 BUY(60%) → SELL(30%) 반복
- 30bars(5시간) → 20분 예측: 시간 규모가 달라 노이즈에 민감

### 개선안

**1-1. 분봉 예측 기간 최적화**

현재 30bars(분) → 20분 후 예측.  
20분은 30분봉 중 2/3에 해당하여 예측 기간이 너무 김.

제안:
```
30bars → 10분 후 예측: 예측 기간 단축 = 신뢰도 ↑
또는 15bars → 10분 후: 더 짧은 데이터로 더 가까운 미래 예측
```

영향: 예측 거리는 짧아지지만 신뢰도는 상승. 단타 모드 취지(빠른 진입/청산)에 부합.

**1-2. 예측 확률 안정화 (Temporal Smoothing)**

현재: 매 사이클 독립적으로 확률 계산 → 확률이 60%→14%로 급변.

제안: 최근 N회 예측 확률의 이동평균 사용
```python
# entry_planner.py 또는 ml/inference.py
# 최근 3회 예측 확률 평균이 threshold 넘을 때만 BUY
prob_history = [61%, 55%, 48%]  # 최근 3회
smoothed_prob = sum(prob_history) / 3  # 54.7%
if smoothed_prob >= buy_threshold:  # BUY
```

장점: 단발성 노이즈에 흔들리지 않음.  
단점: 실제 추세 전환에 3cycle(30초) 지연.

**1-3. Buy Threshold 상향 (60%→65%)**

현재 60% threshold로 BUY 빈도가 높음.  
60%는 AUC 0.674 기준으로 신뢰도가 낮은 구간.

| Threshold | 예상 매수 빈도 | 핑퐁 위험 |
|---|---|---|
| 60% | 높음 | 높음 |
| **65%** | **중간** | **중간** |
| 70% | 낮음 | 낮음 |

**1-4. BUY after N 사이클 안정화**

entry_planner에서 분봉ML BUY 신호가 N회(3회) 연속 동일할 때만 실제 매수 실행.

## 2. KIS 체결내역 동기화 (데이터 무결성)

### 현황
- 서버 재시작 시 pending_buy_trades 버퍼 소멸
- KIS에는 체결되었으나 DB에 미기록
- execute_buy/execute_sell 호출 결과와 DB 불일치

### 개선안

**2-1. 서버 시작 시 KIS 체결내역 조회**

main.py 또는 orchestrator 초기화 시 `inquire_daily_ccld` 호출:
```python
# main.py startup sequence
# KIS 당일 체결내역 조회 → DB와 비교 → 미기록 건 자동 저장
fills = await broker.get_daily_fills()  # 신규 메서드
for fill in fills:
    if not db_has_trade(fill.order_id):
        save_trade_from_kis_fill(fill)
```

**2-2. KIS 잔고 조회 시 포지션 자동 복구**

`refresh_prices()` 또는 `get_positions()` 호출 시  
KIS 포지션과 DB 포지션 불일치 감지 → 자동 기록.

## 3. 중복 sell 방지 강화 (이중 안전장치)

### 현황
- `record_sell`이 success 조건으로 이동됨 (✅ 수정 완료)
- SK이노베이션 5회 중복 sell은 과거 데이터

### 추가 개선

**3-1. Sell 중복 방지 토큰**

`exit_manager.py`에 sell 중복 방지 코드(code, price, quantity) 캐시 추가:
```python
# 동일 조건 sell이 이미 실행되었는지 확인
sell_key = (code, round(sell_price, -2), quantity)
if sell_key in self._recent_sells:
    logger.warning(f"중복 sell 방지: {code} same condition")
    return None
```

## 4. 거래 통계 및 모니터링 강화

### 개선안

**4-1. 매수/매도 사유를 diagnosis에 표시**

현재 diagnosis에 `predictions`만 있고 실제 매수/매도 사유가 없음.  
매수/매도 발생 시 reason을 diagnosis에 포함.

**4-2. 일일 PnL 탭에 거래별 수익률 표시**

각 거래의 buy→sell pair 매칭으로 실현 손익 표시.
