## 1. A-5 ConfirmModal — 위험 액션 확인창

### 내용
`forceSell`, `shutdown`, `stop` 등 위험한 동작을 실행하기 전에
"정말 실행하시겠습니까?" 팝업을 띄워 실수로 인한 사고를 방지

### 리스크 검토

| 항목 | 평가 |
|------|------|
| **리스크 등급** | **ZERO** |
| 위험 요소 | 없음 — 기존 동작을 가로막기만 하고, 추가 권한이나 API 호출 없음 |
| 소요 시간 | **1시간** |
| 영향 범위 | `ConfirmModal.jsx` 신규 + `App.jsx`의 `handleControl`만 수정 |
| 롤백 | App.jsx만 복원 |
| 확인 | 종료 버튼 누르면 확인창 뜨는지, 취소하면 동작 안 하는지 확인 |

### 결론: **리스크 없음, 즉시 진행 가능**

## 2. A-3 DB 엔진 이중화

### 내용
현재 `data/db.py`(전역 싱글톤)와 `live/orchestrator.py`(독자 생성) — 
두 개의 SQLAlchemy 엔진이 각자 DB 연결 풀을 관리.
하나로 통합하여 연결 리소스 낭비와 설정 불일치 해소.

### 리스크 검토

| 항목 | 평가 |
|------|------|
| **리스크 등급** | **MEDIUM** |
| 위험 요소 1 | orchestrator의 DB 초기화 로직 변경 — 세션 생성 방식이 달라짐 |
| 위험 요소 2 | `data/db.py`의 `get_engine()`을 orchestrator가 import하게 되면 **순환 참조** 가능성 |
| 위험 요소 3 | ConfigService도 독자 엔진을 사용 중 — 함께 통합해야 의미 있음 |
| 완화 방안 | 엔진 통합 후 `python main.py` 정상 기동 확인 + `curl /api/trades` API 정상 응답 확인 |
| 소요 시간 | **0.5일** |
| 롤백 | orchestrator.py, api.py 복원 |

### 결론: **MEDIUM 리스크 — 추천 순서 2순위**

## 3. 백테스트 대시보드 탭

### 내용
대시보드에 8번째 탭으로 "백테스트" 추가.
백테스트 실행·진행률·결과 조회를 대시보드에서 가능하게 함.

### 리스크 검토

| 항목 | 평가 |
|------|------|
| **리스크 등급** | **MEDIUM** |
| 위험 요소 1 | **CPU 폭주로 대시보드 느려짐** — 백테스트는 CPU 집약적. 같은 프로세스에서 실행되면 API 응답 지연 |
| → 완화 | `run_in_executor()`로 별도 스레드에서 실행 |
| 위험 요소 2 | **데이터 출처 혼동** — KIS 실데이터 / 합성데이터 fallback을 사용자가 구분 못 함 |
| → 완화 | 결과 화면 상단에 "KIS 실데이터 기반" / "합성 데이터 기반" 뱃지 표시 |
| 위험 요소 3 | **동시 실행** — 실수로 백테스트 2번 누르면 2개 동시 실행 |
| → 완화 | 실행 중 버튼 비활성화 + 진행 상태 표시 |
| 위험 요소 4 | **프론트엔드 복잡도** — 실행 폼 + 결과 화면 + 차트 + 거래내역 4개 영역 |
| → 완화 | Phase 1: 실행+결과요약만. Phase 2: 차트+거래내역 |
| 소요 시간 | **2일** (Phase 1 기준 1일) |
| 롤백 | api.py 롤백 + BacktestPanel.jsx 제거 + App.jsx/Sidebar 원복 |

### 결론: **MEDIUM 리스크 — 적절히 관리 가능**
