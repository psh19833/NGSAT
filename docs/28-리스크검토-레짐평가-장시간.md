# 리스크 검토 보고서 — 장 종료 후 레짐 평가 개선

> 일자: 2026-06-30
> 기준 버그: docs/28-버그-레짐평가-장시간-무관.md

---

## 1. 버그 요약

`orchestrator.run_cycle()`에서 **레짐 평가(Step 3)와 모드 선택(Step 4A), 스크리닝(Step 4B)이 `is_market_hours()` 체크(Step 6)보다 먼저 실행**됨. 장 종료 후에는 마지막 거래일의 스테일 데이터로 레짐이 평가되어 무의미함.

---

## 2. 수정 방안별 리스크 분석

### A안: early return — 장 종료 시 레짐/스크리닝/매수 스킵

**변경 내용**:
`run_cycle()` 최상단(Step 2 직후)에 `is_market_hours()` 체크 추가. 장이 아니면 Steps 3~6을 건너뛰고 Step 7(청산)만 실행하는 별도 분기로 전환.

```python
# Step 2 직후 (위험 체크 통과 후)
market_open = is_market_hours()
if not market_open:
    logger.info("장 운영 시간 아님 — 청산만 실행")
    # Step 7: 청산만 실행
    return await self._run_exit_only(...)
```

| 리스크 | 등급 | 설명 |
|--------|------|------|
| **청산 누락** | **MEDIUM** | early return 전에 청산 로직을 호출해야 함. 현재 청산 루프는 run_cycle 하단에 있고 early return 시 실행되지 않음 → 구조 분리 또는 early return 위치 조정 필요 |
| **대시보드 레짐 공백** | LOW | 장 종료 후 레짐 카드가 "알 수 없음"으로 표시됨. 평소에 보이던 레짐 점수가 사라져서 사용자가 의아해할 수 있음 |
| **계좌/포지션 미갱신** | LOW | 현재 run_cycle이 아니어도 별도 폴링(5초)으로 갱신되므로 영향 없음 |
| **롤백 난이도** | ZERO | `orchestrator.py`만 복구 |

**종합**: **MEDIUM** — 청산 로직 분리 필요. 단순 early return이 아니라 구조 개선이 필요해 작업량 증가.

---

### B안: 장 종료 시 레짐 스킵 + "장 종료" 표시

**변경 내용**:
market_open 체크를 Step 3 직전에 배치. 장이 아니면 레짐 평가/스크리닝을 스킵하고, CycleResult에 `regime_skipped: True` 플래그를 설정. 대시보드에서 이 플래그를 받아 "장 종료 — 마지막 평가: HH:MM" 표시.

```python
market_open = is_market_hours()
if not market_open:
    result.regime_skipped = True
    result.reason = "장 운영 시간 아님 — 레짐/스크리닝 생략, 청산만 실행"
else:
    # Step 3: 레짐 평가 (기존 코드)
    ...
# Step 7: 청산은 항상 실행
```

| 리스크 | 등급 | 설명 |
|--------|------|------|
| **청산 정상 동작** | ZERO | 청산 루프는 run_cycle 하단에 그대로 유지 — early return 없음 |
| **레짐/스크리닝 스킵** | ZERO | 장 종료 시 무의미한 데이터로 평가하는 현재보다 개선됨 |
| **매수 실행 방지** | ZERO | 기존 Step 6의 market_open 체크가 이미 매수를 차단하므로 2중 안전장치 |
| **프론트엔드 수정** | LOW | `RegimeCard.jsx`에서 `regime_skipped` 플래그 처리 추가 필요 |
| **롤백 난이도** | ZERO | `orchestrator.py`만 복구 |

**종합**: **LOW** — 청산에 영향 없고, 현재보다 명확한 동작. 프론트엔드 수정 1~2줄.

---

### C안: evaluated_at_market_hours 플래그 추가 (현행 유지 + 레이블)

**변경 내용**:
레짐 평가는 그대로 진행하되, `RegimeResult`에 `evaluated_during_market_hours: bool` 필드 추가. 대시보드에서 장중/장후 구분 표시.

| 리스크 | 등급 | 설명 |
|--------|------|------|
| **의미없는 데이터 평가 지속** | LOW | 장 종료 후에도 스테일 데이터로 레짐 평가를 계속 수행 — CPU/API 낭비 |
| **프론트엔드 수정** | LOW | RegimeCard.jsx에 "장종료 데이터" 배지 추가 |
| **백엔드 수정** | LOW | RegimeResult + evaluate_regime 시그니처 변경 |
| **근본 해결 아님** | LOW | 데이터가 스테일이라는 사실은 변하지 않음 — 레이블만 추가 |

**종합**: **LOW** — 가장 안전하지만 CPU/API 리소스를 계속 소비. 데이터 품질은 개선되지 않음.

---

## 3. 방안 비교

| 기준 | A안 (early return) | B안 (skip + 표시) | C안 (flag만 추가) |
|------|-------------------|------------------|------------------|
| **리스크** | **MEDIUM** | **LOW** | LOW |
| **청산 안전성** | 구조 변경 필요 | ✅ 보장 | ✅ 보장 |
| **데이터 정합성** | ✅ 완전 해결 | ✅ 완전 해결 | ❌ 스테일 데이터 지속 |
| **프론트 수정** | 없음 | 1~2줄 | 3~5줄 |
| **백엔드 수정** | orcherstrator 구조 변경 | orchestrator 5~10줄 | regime.py + orchestrator 5줄 |
| **작업량** | 30분 | 10분 | 15분 |

## 4. 권장

**B안**을 추천합니다.

| 이유 | 설명 |
|------|------|
| 리스크 LOW | 청산 로직 변경 없음, early return 없음 |
| 데이터 정합성 확보 | 장 종료 후 스테일 데이터로 레짐 평가 안 함 |
| 작업량 최소 | orchestrator.py 5~10줄 + RegimeCard.jsx 1~2줄 |
| 롤백 용이 | 파일 단위 원복 |

진행 여부를 결정해 주시면 B안으로 수정하겠습니다.
