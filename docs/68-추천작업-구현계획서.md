## 3. PC-3: 레짐 기반 동적 손절선 🔴→🟡 High→Medium

### 핵심 설계 원칙
> **"강세장은 넓게, 약세장은 좁게"**

### 수정 사항

#### 3.1 `live/risk.py` — `effective_stop_loss_pct`에 레짐 보정 추가

```python
# 신규 메서드
def regime_adjusted_stop_loss(self, regime_score: float, base_stop_pct: float) -> float:
    \"\"\"레짐 점수 기반 손절선 보정.\"
    if regime_score >= 65:    # Bull
        return min(base_stop_pct + 1.0, self._config.max_stop_loss_pct)  # +1%p 여유
    elif regime_score < 35:   # Bear
        return max(base_stop_pct - 1.0, 1.0)  # -1%p 타이트
    else:                      # Neutral
        return base_stop_pct
```

#### 3.2 `live/exit_manager.py` — `_check_single_exit()` 내 적용

```python
# 기존 (line ~128):
effective_stop = self._risk.effective_stop_loss_pct or position.stop_loss_pct

# 개선:
base_stop = self._risk.effective_stop_loss_pct or position.stop_loss_pct
effective_stop = self._risk.regime_adjusted_stop_loss(ctx.regime_score, base_stop)
```

#### 3.3 기존 `can_extend_stop_loss()` 활용 (선택사항)

이익 발생 포지션에 대해 regime_score가 Bull일 때만 `can_extend_stop_loss()` 호출하여
추가 연장 가능 (기존 max 5% 상한).

### 안전장치
1. **max_stop_loss_pct = 5%** — 절대 상한 (이미 구현)
2. **min_stop_loss_pct = 1%** — 너무 타이트해지지 않도록 하한
3. **레짐 점수 기반** — 레짐 스코어가 낮을수록 손절선 축소 (리스크↓)
4. **bear 모드에서는 확장 금지** — regime_score < 35이면 절대 확장 안 함
5. **기존 모드별 손절선(swing 3%/단타 2%/hold 5%) 위에서 ±1%p 조정**

### 투자 리스크 분석
| 상황 | 보정 전 | 보정 후 | 효과 |
|------|---------|---------|------|
| 강세장(regime ≥ 65) | 3.0% | 4.0% | 추세 조정에 휩쓸리지 않음 → 수익률↑ |
| 약세장(regime < 35) | 3.0% | 2.0% | 손실 조기 확정 → 리스크↓ |
| 중립장 | 3.0% | 3.0% | 변화 없음 |

### 예상 공수
1시간

## 4. #1 분봉ML 자동 재학습 (In-process 방식) 🟢 None

### 문제점

#### 4.2 `ml/inference.py` — 분봉 모델 hot-reload 메서드
```python

    self._minute_model = new_model
```

#### 4.3 `main.py` — `_refresh_data_and_retrain()` 확장
장 마감 감지 시 일봉 모델과 함께 분봉 모델도 재학습:
```python
if config.strategy.ml_minute_auto_retrain:
    minute_model, _ = train_from_minute_data(...)
    if minute_model.is_trained:
        ml_inference.update_minute_model(minute_model)
        minute_model.save()  # models/minute_model.joblib
```

### cron 대신 In-process 선택 이유
| 방식 | 장점 | 단점 |
|------|------|------|
| cron (별도 프로세스) | 격리 | 디스크 I/O + 프로세스 관리 + 상태 공유 복잡 |
| in-process (현재 방식 확장) | 메모리 직접 교체, 기존 로직 재사용 | 메모리 사용 (훈련 중) |

in-process 방식이 기존 인프라(장 마감 감지, 주기적 데이터 갱신)를 그대로 활용 가능.

### 리스크
- 🟢 **없음**. 장 마감 후 실행, 실패해도 기존 모델 유지
- 훈련 중 메모리/CPU 사용 → 장 마감 후이므로 영향 없음
