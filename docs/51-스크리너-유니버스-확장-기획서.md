## 1. 문제 정의

### 1.1 현황

스크리너 통과 종목이 **1종목**에 불과한 상황.

| 항목 | 현재 값 | 비고 |
|:----|:-------:|:-----|
| 스크리닝 대상 종목 | ~40개 | `_universe_cache` 기준 |
| 스크리닝 통과 | 1개 | 중립장 min_score=60 |
| 통과율 | ~2.5% | |
| UniverseManager 관리 종목 | 100개 (active 40 + reserve 60) | 효율적 관리 중 |
| 실제 스크리닝 반영 | active 40개만 | reserve 60개 미반영 |

### 1.2 근본 원인

**두 계층 간 유니버스 캐시 동기화 불일치**가 핵심 원인이다.

```
RealDataProvider.__init__()       → codes = DEFAULT_UNIVERSE_CODES (40종목)
RealDataProvider.load()           → _universe_cache = 40종목 저장
UniverseManager.initialize()       → 100종목 관리 시작
    _load_daily_data(codes[:100])  → provider._universe_cache에 추가 시도
main.py._manage_universe()         → pool_codes(100) ∩ _universe_cache(40) = 최대 40종목
                                    → reserve 60종목은 스크리닝에서 제외됨
```

`_manage_universe()`는 `data_provider._universe_cache`에서 pool_codes에 해당하는 종목만 필터링해 반환한다(line 414-417). 그런데 `_universe_cache`에는 최초 `RealDataProvider.load()` 시점에 하드코딩된 40종목만 확실히 존재하고, `UniverseManager._load_daily_data()`로 추가 로드된 60종목은 rate limit/KIS API 제한으로 누락되거나 `_universe_cache`에 온전히 반영되지 않는다.

### 1.3 부차 요인

| 요인 | 영향 | 상세 |
|:----|:----:|:-----|
| Refresh 간격 50분 | 중 | 교체된 reserve 종목 데이터가 스크리닝에 반영까지 최대 50분 지연 |
| 스크리너 60일 기준 | 중 | 신규 편입 종목이 데이터 부족으로 탈락 가능 |
| 중립장 min_score=60 | 하 | 적정 수준, 점수체계 변경 불필요 |

## 2. 개선 방안

점수체계/임계값은 **변경하지 않고**, 더 많은 종목이 스크리닝 대상이 되도록 유니버스 데이터 흐름을 개선한다.

### 2.1 A안: 유니버스 캐시 동기화 (필수)

**목표**: reserve 60종목의 일봉 데이터가 스크리닝에 사용되도록 보장한다.

#### 변경 사항

##### 2.1.1 `UniverseManager` — reserve 종목 데이터 자체 보관

`UniverseManager`가 reserve 종목의 일봉 데이터를 별도로 캐싱한다.

```python
# universe_manager.py —新增 필드
class UniverseManager:
    def __init__(self):
        self.active: dict[str, ScoredStock] = {}
        self.reserve: dict[str, ScoredStock] = {}
        self._reserve_prices: dict[str, list[PriceData]] = {}  # reserve 일봉 캐시 (新增)
        self._active_prices: dict[str, list[PriceData]] = {}    # active 일봉 캐시 (新增)
```

##### 2.1.2 `_load_daily_data()` — 일봉 데이터를 자체 캐시에도 저장

`_load_daily_data()`가 provider._universe_cache에만 저장하지 않고, `UniverseManager` 자체 캐시(`_active_prices`, `_reserve_prices`)에도 저장한다.

```python
async def _load_daily_data(self, codes: list[str], provider: Any) -> None:
    # ...기존 로직...
    if prices:
        # provider._universe_cache에도 추가 (기존 유지)
        # + 자체 캐시에도 저장
        if code in self.active:
            self._active_prices[code] = prices
        else:
            self._reserve_prices[code] = prices
```

##### 2.1.3 `_manage_universe()` — reserve 데이터 포함 반환

`main.py._manage_universe()`가 reserve 종목의 일봉 데이터를 `UniverseManager` 캐시에서 조회해 반환한다.

```python
async def _manage_universe() -> list:
    # ...기존 로직...
    pool_codes = set(um.active.keys()) | set(um.reserve.keys()) if um.initialized else None
    result = []
    
    # 1) provider._universe_cache에서 조회 (기존 active 종목)
    if pool_codes and data_provider._universe_cache:
        include_codes = pool_codes | orchestrator._last_held_codes
        for info, prices in data_provider._universe_cache:
            if info.code in include_codes:
                result.append((info, prices))
    
    # 2) reserve 종목 중 _universe_cache에 없는 것 → um._reserve_prices에서 보충 (新增)
    found = {info.code for info, _ in result}
    missing = pool_codes - found if pool_codes else set()
    for code in missing:
        prices = um._reserve_prices.get(code)
        if prices:
            # StockInfo 구성 (name/market은 universe_manager에서 관리)
            info = StockInfo(code=code, name=um.reserve[code].name if code in um.reserve else "",
                             market=_infer_market(code), product_type="stock")
            result.append((info, prices))
    
    return result
```

##### 2.1.4 `swap()` — 교체 시 캐시 이동

`UniverseManager.swap()`에서 active→reserve 이동 시 `_active_prices[code]` → `_reserve_prices[code]`로 데이터도 함께 이동한다.

```python
# swap() 내부, 교체 실행 부분
for s in to_add:
    self.active[s.code] = s
    if s.code in self._reserve_prices:        # 新增
        self._active_prices[s.code] = self._reserve_prices.pop(s.code)
for s in to_remove:
    self.active.pop(s.code, None)
    self.reserve[s.code] = s
    if s.code in self._active_prices:         # 新增
        self._reserve_prices[s.code] = self._active_prices.pop(s.code)
```

#### 기대 효과

| 항목 | Before | After |
|:----|:------:|:-----:|
| 스크리닝 대상 종목 | ~40개 | ~100개 (2.5배 ↑) |
| 통과 종목 (추정) | 1개 | 2~4개 |
| 추가 API 호출 | 없음 | 없음 (기존 로드된 데이터 재활용) |

#### 리스크

| 리스크 | 확률 | 대응 |
|:-------|:----:|:-----|
| reserve 종목의 일봉 데이터 미존재 | 중 | `_load_daily_data` 실패 시 skip, 다음 refresh에서 자동 복구 |
| `StockInfo` 메타정보 부정확 | 하 | 종목코드로 market 추론, name은 빈값으로 두어도 스크리닝 점수에 영향 없음 |
| 메모리 사용 증가 | 하 | 60종목 × 250일 × 6필드 ≈ 360KB — 무시 가능 |

### 2.2 C안: Refresh 간격 단축 (보조)

**목표**: 교체된 reserve 종목의 일봉 데이터가 스크리닝에 신속히 반영되도록 refresh 주기를 단축한다.

#### 변경 사항

##### 2.2.1 `REFRESH_INTERVAL` 300→60

`main.py` line 285:

```python
# 변경 전
REFRESH_INTERVAL = 300  # 300사이클(50분)마다 refresh

# 변경 후
REFRESH_INTERVAL = 60   # 60사이클(10분)마다 refresh
```

(`tick_interval=10s` 기준: 300×10=3,000초=50분 → 60×10=600초=10분)

##### 2.2.2 `refresh_prices()` rate limit 보호 확인

`refresh_prices()`는 40종목 × 0.1초 sleep = 4초 소요. 10분 간격이면 시간당 6회 = 240회 API 호출 = **KIS rate limit(초당 20회)에 여유 있음**.

다만 A안 적용 후 100종목으로 확장되면 `refresh_prices`도 100종목을 갱신해야 한다. 이 경우:

```python
# refresh_prices() — 100종목 버전으로 확장 (A안과 연계)
# rate limit 0.1초 간격 유지, 100종목 × 0.1초 = 10초
```

#### 기대 효과

| 항목 | Before | After |
|:----|:------:|:-----:|
| refresh 주기 | 50분 (300사이클) | **10분 (60사이클)** |
| reserve 교체→반영 지연 | 최대 50분 | 최대 10분 |
| 시간당 refresh 횟수 | 1.2회 | 6회 |
| 시간당 API 호출량 | ~48회 | ~240회 (KIS 제한 내) |

#### 리스크

| 리스크 | 확률 | 대응 |
|:-------|:----:|:-----|
| KIS Rate Limit 초과 | 낮음 | 10분 간격 × 100종목 × 0.1초 = 10초, 초당 10회 이하로 안정적 |
| refresh 중 사이클 지연 | 낮음 | refresh는 10분에 1회만 실행, 나머지 사이클은 정상 동작 |

| 안 | 효과 | 복잡도 | 리스크 |
|:-:|:----:|:------:|:------:|
