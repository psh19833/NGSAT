## 1. 현재 ML 시스템 개요

### 1.1 현재 운영 중인 모델 (Production)

| 항목 | 값 | 비고 |
|------|------|------|
| **모델 타입** | RandomForestClassifier | scikit-learn 앙상블 100트리 |

## 2. 현재 모델 성능 진단

### 2.1 강점 (Strengths)

| 항목 | 평가 |
|------|------|
| ✅ 자동 재학습 | 매 사이클마다 `auto_retrain()`으로 최신 데이터 반영 |
| ✅ 무결성 검증 | SHA-256 해시로 모델 파일 변조 감지 |
| ✅ 타임시리즈 분할 | TimeSeriesSplit으로 과거→미래 순서 보존 평가 |
| ✅ 불균형 대응 | class_weight='balanced' 사용 |
| ✅ 다양성 | 5개 모델 타입 지원, Optuna 튜닝 내장 |
| ✅ 분봉 단타 지원 | 2단계(일봉→분봉) 하이브리드 아키텍처 구현 완료 |

### 2.2 개선 필요 사항 (Weaknesses)

| 항목 | 현재 | 문제점 |
|------|------|--------|
| ❌ AUC 낮음 | ~0.473 (무작위 수준) | 0.5가 무작위 예측 — 모델이 거의 학습되지 않음 |
| ❌ 양성비율 극도 낮음 | ~30.9% | 클래스 불균형 심각, 타겟 정의(3일/2%) 재검토 필요 |
| ❌ 정밀도 0% | precision=0.0 | 모든 양성 예측이 틀림 — 신뢰도 0 |
| ❌ 펀더멘털 피처 미사용 | 외인/기관/PER/PBR/EPS = 항상 0 | 실제 데이터 소스 미연결 |
| ❌ 단일 모델 | 하나의 모델만 사용 | 앙상블/스태킹/투표 없음 |
| ❌ feature_importance 미활용 | 로그만 출력 | 실제 트레이딩 결정에 반영 안 됨 |
| ❌ 모델 선택 로직 없음 | model_type 고정 | 시장 상황별 최적 모델 자동 선택 X |
| ❌ 백테스트 부족 | 학습/평가만 | 실제 매매 시뮬레이션 미검증 |

### 2.3 기회 (Opportunities)

| 기회 | 설명 |
|------|------|
| 🎯 **GradientBoosting 즉시 전환 가능** | 추가 설치 없이 `model_type="gradient_boosting"` 만 변경 |
| 🎯 **외인/기관 순매수 데이터 연결** | KIS API `inquire_investor` 엔드포인트로 실데이터 수집 |
| 🎯 **Optuna 튜닝 자동화 가능** | 현재 수동 호출 중 → `auto_retrain()` 내에 통합 가능 |
| 🎯 **XGBoost/LightGBM 설치만 하면 사용 가능** | pip install + model_type 변경 = 즉시 적용 |
| 🎯 **분봉 단타 모델 활성화** | 코드는 완료, 모델 학습만 진행하면 됨 |
| 🎯 **KIS WebSocket 실시간 데이터 활용** | 연결 안정화 시 실시간 업데이트 가능 |

### 2.4 위협 (Threats)

| 위협 | 영향 |
|------|------|
| ⚠️ **데이터 부족으로 안정적 학습 어려움** | 275개 샘플 (6/29 기준) — 딥러닝에 턱없이 부족 |
| ⚠️ **KIS API Rate Limit** | 고빈도 데이터 수집 시 호출 제한에 막힐 위험 |
| ⚠️ **WebSocket 불안정** | 분봉 단타 모델 = 실시간 데이터 필수 |
| ⚠️ **과적합 위험** | 샘플 적고 피처 많은 구조, max_depth 제한 있지만 불충분 |

## 3. 개선 방안 (선택지)

### 선택지 A: 현재 모델 튜닝 (리스크 LOW)

**내용:** RandomForest 유지, 하이퍼파라미터 최적화 + 피처 개선

| 작업 | 예상 시간 | 리스크 |
|------|-----------|--------|
| A1. Optuna 자동 튜닝을 `auto_retrain`에 통합 | 2h | LOW |
| A2. forward_days (3→5) / threshold (2%→3%) 조정 실험 | 1h | LOW |
| A3. 불필요 피처 제거 (외인/기관/PER/PBR/EPS = 상수 제외) | 0.5h | LOW |
| A4. Early Stopping 조건 추가 (AUC < 0.5면 모델 교체 안 함) | 0.5h | LOW |

**총 예상:** 4시간
**예상 효과:** AUC 0.47 → 0.55~0.60

**위험:**
- LOW: 피처 제거/파라미터 변경으로 인한 회귀는 `auto_retrain()`이 기존 모델 유지로 자동 방어
- LOW: forward_days 변경 시 기존 저장 모델과 호환성 문제 없음 (predict_proba에서 자동 패딩)

---

### 선택지 B: GradientBoosting 전환 (리스크 LOW-MEDIUM)

**내용:** 현재 모델을 `gradient_boosting`으로 전환 + A 선택지 병행

| 작업 | 예상 시간 | 리스크 |
|------|-----------|--------|
| B1. model_type="gradient_boosting" 변경 | 0.1h | LOW |
| B2. auto_retrain으로 1회 학습 → AUC 비교 | 0.5h | LOW |
| B3. 결과 좋으면 적용, 나쁘면 롤백 | 0.5h | LOW |
| (A1~A4 병행) | 4h | LOW |

**총 예상:** 5시간
**예상 효과:** AUC 0.47 → 0.55~0.65 (HistGradientBoosting은 일반적으로 RF보다 우수)

**위험:**
- MEDIUM: HistGradientBoosting은 `coef_`나 `feature_importances_`가 없어 permutation importance 사용 (이미 구현됨)
- LOW: 학습 시간이 RF보다 약간 더 걸림 (샘플 275개에서는 미미)

---

### 선택지 C: 외인/기관 순매수 데이터 연결 (리스크 MEDIUM)

**내용:** KIS API로 진짜 외인/기관 순매수 데이터를 가져와 7개 피처 활성화

| 작업 | 예상 시간 | 리스크 |
|------|-----------|--------|
| C1. KIS `inquire_investor` 엔드포인트 정의 | 0.5h | LOW |
| C2. adapter.py에 investor 순매수 조회 메서드 추가 | 1h | LOW |
| C3. feature builder에서 KIS 데이터 연결 | 1.5h | LOW |
| C4. Rate Limit 대응 — 캐싱/배치 처리 | 0.5h | MEDIUM |
| C5. C~A 병행 테스트 | 1h | LOW |

**총 예상:** 4.5시간
**예상 효과:** AUC 0.55~0.65 (펀더멘털 데이터 추가로 정보량 증가)

**위험:**
- MEDIUM: KIS API Rate Limit 문제 재발 가능 (inquire_investor도 호출 제한 있음)
- LOW: 외인/기관 데이터가 없어도 기존처럼 0 채움으로 fallback 가능

---

### 선택지 D: 앙상블/스태킹 모델 (리스크 HIGH)

**내용:** 3~5개 모델을 동시에 학습하고 투표/VotingClassifier로 통합

| 작업 | 예상 시간 | 리스크 |
|------|-----------|--------|
| D1. VotingClassifier 래퍼 구현 | 2h | LOW |
| D2. 개별 모델 병렬 학습 파이프라인 | 1h | LOW |
| D3. 저장/로드 포맷 확장 | 1h | LOW |
| D4. 추론 속도 최적화 | 0.5h | MEDIUM |
| D5. 1~2주 실운영 모니터링 | - | HIGH |

**총 예상:** 4.5h + 모니터링 2주
**예상 효과:** AUC 0.55~0.70

**위험:**
- HIGH: 앙상블 추론 시간 = 단일 모델 × N배 — 10초 사이클에 영향
- HIGH: 모델 저장 크기 N배 증가 (현재 1.8MB × 5 = 9MB)
- MEDIUM: 개별 모델 중 일부가 나쁘면 전체 앙상블 성능 저하
- HIGH: 디버깅 복잡도 급증 (어느 모델이 왜 틀렸는지 추적 어려움)

## 4. 권장 실행 계획

### ➡️ 권장: **선택지 B + A 병행 (1차) → C (2차)**

근거: 가장 적은 리스크로 가장 빠른 성능 개선 가능. GradientBoosting은 추가 설치 없음.

### 1차 (이번 스프린트, ~5시간)

```text
Phase 1.1 (1h) — 튜닝 인프라
  ├ auto_tune()을 auto_retrain()에 통합
  ├ 튜닝 주기: 20사이클마다 1회 자동 튜닝
  └ 리스크: LOW — 튜닝 실패해도 기존 모델 유지

Phase 1.2 (1h) — GradientBoosting 평가
  ├ model_type="gradient_boosting"으로 전환 학습
  ├ AUC/RF-AUC 비교
  └ 리스크: LOW — 결과 나쁘면 model_type 한 줄만 복구

Phase 1.3 (1h) — 타겟 튜닝
  ├ forward_days: 3/5/7 비교
  ├ forward_threshold: 1.5%/2%/3% 그리드서치
  └ 리스크: LOW — auto_retrain이 자동 방어

Phase 1.4 (2h) — 펀더멘털 피처 연결 (C1~C3)
  ├ KIS inquire_investor 연동
  ├ feature builder에 실데이터 연결
  └ 리스크: MEDIUM — KIS Rate Limit, fallback=0
```

### 2차 (다음 스프린트, 조건부)

```text
Phase 2 — 분봉 단타 모델 활성화 (risks MEDIUM)
  ├ minute_data 수집 파이프라인 안정화
  ├ KIS WebSocket 연결 개선 (별도 이슈)
  ├ 분봉 모델 학습 및 A/B 테스트
  └ 조건: WebSocket 안정화 후 진행
```

## 5. 롤백 계획

| 변경 사항 | 롤백 방법 | 복구 시간 |
|-----------|-----------|-----------|
| model_type 변경 | `auto_retrain()`이 AUC 비교 → 나쁘면 기존 유지 | 자동 |
| auto_tune() 통합 | 이전 버전 코드로 복구 | 10분 |
| 펀더멘털 피처 | feature_importance 0으로 자연 소멸 | 자동 (재학습 시) |
| 앙상블 도입 | 단일 모델로 폴백 설정 | 5분 |

> **결론:** 1차 개선으로 AUC 0.47 → 0.55~0.65 목표. 가장 큰 병목은 현재 타겟 정의(3일/2%)와 데이터 부족. GradientBoosting 전환 + 타겟 튜닝으로 즉시 효과를 볼 수 있음.
