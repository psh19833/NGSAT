### H-1. risk.py — falsy 함정 (stop_loss_pct or default)

| 항목 | 내용 |
|------|------|
| **파일** | `live/risk.py` — `can_extend_stop_loss()` |
| **현황** | `position.stop_loss_pct or self._config.default_stop_loss_pct` — Python에서 `0.0 or 3.0 = 3.0`으로 평가됨. 손절선이 명시적으로 0%면 `False`로 처리되어 의도와 다르게 기본값(3.0) 사용 |
| **영향** | 즉시 손절(0%) 정책 무시. 손절선 연장 로직 오작동 |
| **처리** | `or` 대신 `is not None` 명시적 체크로 변경 |
| **리스크** | 🔴 ZERO — 조건식만 변경, 로직 변경 없음 |
| **소요** | 5분 |

```python
# 변경 전
if new_stop_loss_pct <= (position.stop_loss_pct or self._config.default_stop_loss_pct):

# 변경 후
current_stop = (
    position.stop_loss_pct
    if position.stop_loss_pct is not None
    else self._config.default_stop_loss_pct
)
if new_stop_loss_pct <= current_stop:
```

| **소요** | 5분 |

# 변경 후
if last_error is None:
    raise BrokerError(
        f"주문 제출 실패: {code} {side.value} — "
        f"max_retries({max_retries})가 0 이하입니다"

### H-3. main.py — circuit breaker 패턴

| 항목 | 내용 |
|------|------|
| **파일** | `main.py` — `trading_loop()` except 블록 |
| **현황** | 연속 에러 발생 시 단순 로깅 + sleep 후 재시도. 동일 에러(KIS 인증 실패, DB 단절)가 반복되면 로그 폭증 + Rate Limit 추가 위반 |
| **영향** | KIS Rate Limit 악순환, 로그 디스크 풀, 무한 재시도로 인한 리소스 고갈 |
| **처리** | 연속 에러 카운터 + 지수 백오프 + 임계치 도달 시 루프 중단 |
| **리스크** | 🟡 LOW — 새로운 전역 변수(`_consecutive_errors`) 도입. 중단 임계치(5회)는 안전장치로 과도한 중단 방지용 |
| **소요** | 15분 |

```python
# 변경 후 구조
_consecutive_errors = 0
MAX_CONSECUTIVE_ERRORS = 5

except Exception as e:
    _consecutive_errors += 1
    if _consecutive_errors >= MAX_CONSECUTIVE_ERRORS:
        logger.critical("연속 에러 5회 — 매매 루프 중단")
        break
    backoff = min(tick_interval * (2 ** _consecutive_errors), 300)
    await asyncio.sleep(backoff)
# 성공 사이클마다 _consecutive_errors = 0 리셋
```
