## Phase 1: 치명 버그 수정 (즉시, 1~2일)

### 1-1. auto_tune() 모델 복구
- **파일:** `ml/training/trainer.py` (440행)
- **문제:** `self._model = study.best_params` — dict를 모델로 할당, predict_proba() 시 AttributeError
- **수정:** best_params로 새 모델 생성 → `fit(X, y)` 호출 → `self._model` 할당
- **위험:** 없음 (현재 auto_tune() 실행 시 모델 깨짐 → 이 코드만 없애도 동작 개선됨)
- **난이도:** ★☆☆☆☆ (30분)

```python
# Before
self._model = study.best_params
self._is_trained = True

# After
best_model = self._create_model()  # 내부 모델 생성 메서드 필요
best_model.set_params(**study.best_params)
best_model.fit(X_train, y_train)   # X_train/y_train은 study 내부에서 사용한 데이터
self._model = best_model
self._is_trained = True
```

### 1-2. main.py 실시간 데이터 재조회
- **파일:** `main.py` (trading_loop 함수)
- **문제:** while 루프 진입 전 1회만 `provider.load()` 호출, 이후 동일 데이터 반복 사용
- **수정:** 매 사이클마다 `provider.load()` (또는 증분 업데이트) 호출하도록 변경
- **위험:** API 호출량 증가 (분당 6회 → KIS 제한 20회/초 이내면 OK)
- **난이도:** ★★☆☆☆ (1일)

```python
# Before
universe, index_prices = await data_provider.load()
while True:
    if orchestrator.controller.is_running:
        result = await orchestrator.run_cycle(index_prices, universe)

# After
while True:
    if orchestrator.controller.is_running:
        # 매 사이클 최신 데이터 갱신
        latest_prices = await data_provider.load_latest()
        if latest_prices:
            index_prices, universe = latest_prices
        result = await orchestrator.run_cycle(index_prices, universe)
```

### 1-3. 단타 손익비 재설정
- **파일:** `core/config.py` (123, 133행)
- **문제:** 타겟 +0.5% / 손절 -1.5% = 손익비 1:3, 매매비용 포함 시 승률 80%+ 필요
- **수정:** 타겟을 +1.0%~+1.5%로 상향, 손절 -1.0%로 하향 (손익비 1:1~1:1.5)
- **위험:** 단타 진입 횟수 감소 가능, AUC 75% 기준으로 승률 60%면 기대수익 양수로 개선
- **난이도:** ★☆☆☆☆ (30분, 파라미터 수정 + 백테스트 재확인)

| 설정 | 타겟 | 손절 | 손익비 | 필요 승률 (비용 전) | 필요 승률 (비용 후 0.35%) |
|------|:----:|:----:|:------:|:-------------------:|:------------------------:|
| **현재** | +0.5% | -1.5% | 1:3.0 | 75.0% | 80.5% |
| **A안** | +1.0% | -1.0% | 1:1.0 | 50.0% | 55.5% |
| **B안 (추천)** | +1.0% | -0.8% | 1:0.8 | 44.4% | 50.2% |
| **C안** | +1.5% | -1.0% | 1:0.67 | 40.0% | 46.2% |

## Phase 2: 설계 결함 개선 (우선순위, 2~4일)

### 2-1. DB 의존성 주입 및 생명주기 정비
- **파일:** `live/orchestrator.py` (118-131행)
- **문제점 3가지:**
  1. SQLite 경로 하드코딩 — 설정의 PostgreSQL URL 무시
  2. `Session` 인스턴스 변수 보관 — 스레드 안전성 위험
  3. `commit()` 호출 시 예외 롤백 없음
- **수정안 A (단기):** 설정에서 DB URL 읽도록 변경 + `with session.begin()` 패턴
- **수정안 B (장기):** DB 세션 팩토리를 orchestrator 외부에서 주입받도록 설계 변경
- **난이도:** ★★★☆☆ (1~2일)

### 2-2. 글로벌 config → DI로 전환
- **파일:** `strategy/regime.py`, `strategy/screener.py`, `strategy/mode_selector.py`
- **문제:** 3개 모듈이 공유 `_strategy_config` 글로벌 변수 사용, 런타임 교체 시 경쟁 조건
- **수정:** 각 함수 인자로 config를 명시적 전달. 단기: frozenset/ImmutableConfig로 교체
- **난이도:** ★★★☆☆ (1일)

### 2-3. mode_selector 모듈 레벨 상수 제거
- **파일:** `strategy/mode_selector.py` (58-59행)
- **문제:** `HIGH_VOLATILITY_ATR_PCT` 등이 임포트 시점에 고정되어 잘못된 값 보유
- **수정:** 상수 선언 제거, 본문에서 `_get_mode_config()` 직접 호출로 충분
- **난이도:** ★☆☆☆☆ (30분)

### 2-4. 포지션 집중 리스크 제어
- **파일:** `live/orchestrator.py` (285행 부근)
- **문제:** 포지션 당 10%지만 최대 종목 수, 업종 한도, 단일 종목 최대 비중 없음
- **수정:** Config에 `max_holdings`, `max_sector_exposure_pct` 필드 추가 → 진입 시 체크
- **난이도:** ★★☆☆☆ (1일)

## Phase 3: 코드 품질 개선 (여유 있을 때, 2~3일)

### 3-1. 예외 처리 설계 재검토
- **파일:** `strategy/screener.py` (228행), `live/orchestrator.py` (449행) 외
- **문제:** `except Exception: continue` — 패턴 감지 실패 시 점수 왜곡
  `except (NotImplementedError, Exception)` — NotImplementedError 중복 포착
- **수정:** 개별 예외 타입 명시 + 최소 `logger.warning` 기록

### 3-2. orchestrator 중복 return 제거
- **파일:** `live/orchestrator.py` (432행)
- **난이도:** ★☆☆☆☆ (5분) — 단순 데드코드 제거

### 3-3. EMA 표준화
- **파일:** `strategy/indicators.py` (55행)
- **문제:** `result[0] = arr[0]` — 표준 EMA는 SMA를 시드로 사용
- **수정:** 초기값을 `np.mean(arr[:period])`로 대체 (또는 주석으로 명시)
- **난이도:** ★☆☆☆☆ (30분)

### 3-4. RSI 인덱스 정렬
- **파일:** `strategy/indicators.py` (89-91행)
- **문제:** `gains[i-1]`이 `i=period`에서 이미 SMA에 포함된 값을 중복 계산
- **수정:** 루프를 `range(period+1, len(arr))`로 시작, 첫 RSI는 SMA 그대로 사용
- **난이도:** ★☆☆☆☆ (30분)

### 3-5. dashboard .env 직접 수정 개선
- **파일:** `dashboard/backend/api.py`
- **문제:** PUT `/api/strategy/config`가 .env 파일 직접 수정 → 동시 쓰기 손상 위험
- **수정안 A (단기):** 파일 락(`fcntl.flock`) 추가
- **수정안 B (장기):** 설정 저장소(DB 또는 별도 config 서비스)로 이관
- **난이도:** ★★☆☆☆ (1일)
