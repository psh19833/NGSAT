## 2. 근본 원인 분석

### 2.1 KIS 랭킹 API ETN/ETF 제외 파라미터 미동작

랭킹 API 3종(거래량순위/체결강도/등락률)에 `FID_TRGT_EXLS_CLS_CODE="0000001100"` (ETF/ETN 제외) 파라미터가 설정되어 있으나, 실제로는 ETN/ETF가 그대로 반환됨.

실제 로그에서 확인된 ETN 코드:
```
일봉 조회: 0030R0 100개   ← 문자 포함 = ETN
일봉 조회: 03481K 100개   ← ETN  
일봉 조회: 0068Y0 100개   ← ETN
```

### 2.2 이중 필터링으로 reserve 소멸

랭킹 API 반환 200+코드 → `_get_stock_type()` 필터로 정상 주식만 통과

```
랭킹 API 200+코드
  → 점수 계산 (all codes, 200개)
  → _get_stock_type() == "stock" 필터 (160개 ETN/ETF 제거)
  → scored = 40개 (정상 주식만 남음)
  → active[:40] = 40, reserve[40:] = 0
```

**reserve가 0이어서 A안 무력화.** `_manage_universe()`가 아무리 reserve fallback을 시도해도 보충할 대상이 없음.

### 2.3 `_score_candidates`의 screener_lookup 제한

`_score_candidates()`는 `provider._universe_cache`(초기 40종목)만 스크리너 점수 계산에 사용. 랭킹 API 전용 종목(ETN 포함)은 screener_score=50.0 고정. 이 자체는 점수만 낮출 뿐 reserve 생성에 직접 영향 없음.

## 3. F안: 하이브리드 유니버스

### 3.1 개념

랭킹 API에 **100% 의존하지 않고**, 안정적인 베이스 유니버스(`DEFAULT_UNIVERSE_CODES` 40종목)를 보장한 상태에서 랭킹 API로 보충하는 하이브리드 방식.

```
기존: all_codes = 랭킹API 결과 (ETN/ETF 포함 200개)
F안: all_codes = DEFAULT_UNIVERSE_CODES (40) + 랭킹API 결과 중 주식만 (최대 100)
```

### 3.2 변경 사항

#### 3.2.1 `UniverseManager.initialize()` — 유니버스 구성 방식 변경

**변경 전** (`universe_manager.py:93-105`):
```python
all_codes = self._merge_rankings(volume_rank, volume_power, fluctuation)
# all_codes: 랭킹 API에만 의존 (ETN/ETF 다량 포함)
```

**변경 후**:
```python
ranking_codes = self._merge_rankings(volume_rank, volume_power, fluctuation)
# 베이스: DEFAULT_UNIVERSE_CODES 40종목 보장
from data.real_data_provider import DEFAULT_UNIVERSE_CODES
base_codes = list(DEFAULT_UNIVERSE_CODES)
# 합병: base 우선, ranking으로 보충
all_codes = list(dict.fromkeys(base_codes + ranking_codes))  # 중복 제거, 순서 유지
```

#### 3.2.2 `_get_stock_type` 필터 유지

랭킹 API가 ETN/ETF를 완전히 제외하지 못하므로 `_get_stock_type` 필터는 **유지**. 다만 base 40종목은 이미 주식으로 확인되었으므로 필터 통과가 보장됨.

#### 3.2.3 reserve 생성 보장

Base 40종목 + 랭킹 보충 (ETN 제외 후 평균 20~40종목) = 총 60~80종목

```
scored (60~80종목)
  → active[:40] = 40
  → reserve[40:] = 20~40 ✅
```

### 3.3 기대 효과

| 항목 | Before (A+C안) | After (F안) |
|:----|:-------------:|:-----------:|
| 유니버스 총 종목 | 40 | 60~80 |
| reserve | 0 | 20~40 |
| 스크리닝 대상 | ~37개 | ~60개 |
| 스크리닝 통과(추정) | 1개 | 2~4개 |
| 하드코딩 vs 동적 | 동적 | **하이브리드** (40 고정 + 동적 보충) |

## 4. 리스크 분석

### 4.1 기술 리스크

| # | 리스크 | 영향 | 확률 | 대응 |
|:-:|:-------|:----:|:----:|:-----|
| R1 | `DEFAULT_UNIVERSE_CODES`에 상장폐지/거래정지 종목 포함 | 스크리닝 시 0점 처리, 진입 안 됨 | 낮음 | 연 1~2회 리스트 점검 |
| R2 | ranking API가 항상 같은 40종목만 반환 (base와 동일) | reserve=0 (A안과 동일) | 중 | 3개 API + 등락률 100종목 요청으로 다양성 확보 |
| R3 | base 40종목 중 랭킹 하위 종목이 active에 계속 잔류 | 저점수 종목이 active 차지 | 낮음 | swap()에서 주기적 교체 |
| R4 | `all_codes` 중복 제거 시 `dict.fromkeys`로 인한 정렬 손실 | base 순서 보존되나 ranking 순서 섞임 | 낮음 | 성능에 영향 없음 |

### 4.2 운영 리스크

| # | 리스크 | 영향 | 확률 | 대응 |
|:-:|:-------|:----:|:----:|:-----|
| R5 | KIS API rate limit 악화로 ranking API 실패 | `_last_rank_cache` fallback, base 40만 유지 | 중 | base 40이 항상 보장되므로 최소 40종목 유지 |
| R6 | 서버 재시작 시 캐시 초기화로 첫 사이클 reserve=0 | 첫 5분간 reserve 없음 | 중 | swap() 5분 후 자동 복구 |
| R7 | ETF/ETN 필터가 ETN을 제대로 식별 못 함 | ETN이 유니버스에 편입되어 스크리닝 통과 가능 | 낮음 | `run_cycle()`에서 재차 필터링 (ETN 제거 로직 유지) |

### 4.3 Rollback 방안

| 변경 | Rollback |
|:-----|:---------|
| `initialize()`의 base 병합 로직 | 병합 코드 1개 if문 제거 |
| `_get_stock_type` 필터 유지 | 유지 (기존과 동일) — rollback 불필요 |
