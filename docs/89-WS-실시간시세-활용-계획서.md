## 1. 현재 상황 분석

### WebSocket이 이미 하는 일

현재 KIS WebSocket 실시간 체결가(`H0UCNT0`)는 `on_price` 콜백으로 `_universe_cache`의 **마지막 일봉을 실시간 갱신 중**:

```
WS 체결가 수신 → on_price(code, price, high, low, open, volume, ts)
  → _universe_cache[code].prices[-1] 업데이트
    → high = max(기존고가, 현재가)
    → low = min(기존저가, 현재가)
    → close = 현재가
    → volume = 누적거래량
```

즉, **REST `refresh_prices()`의 intraday 기능을 WS가 이미 수행 중**.

### 현재 비효율

| 항목 | 상태 |
|------|------|
| WS 실시간 last bar 갱신 | **✅ 이미 동작 중** |
| REST refresh_prices (10분) | **❌ WS와 동일한 작업 중복** |
| WS 미구독 종목 | KOSPI 지수(`0001`) — REST만 가능 |
| 장 시작 첫 1회 | REST 필요 (init load) |
| 장 마감 후 | WS 중단, REST만 가능 |

## 2. 개선 방안

**핵심 아이디어:** `refresh_prices()`에서 WS가 실시간 갱신 중인 intraday 종목은 REST 호출을 생략.

### 변경 전 (현재)
```
refresh_prices() 실행 (10분마다):
  for 100개 종목:
    REST get_price_history(code)  ← 100회 API 호출
```

### 변경 후
```
refresh_prices() 실행 (10분마다):
  if WS connected AND today == 장중:
    for 100개 종목:
      if code in WS_구독목록 AND prices[-1].date == 오늘:
        SKIP (WS가 이미 갱신 중)  ← 0회 API 호출 (intraday)
      else:
        REST get_price_history(code)  ← 소수만 호출
  else (WS 미연결 or 장외):
    for 100개 종목:
      REST get_price_history(code)  ← 기존대로 100회
```

### 변경 범위

**파일:** `data/real_data_provider.py` — `refresh_prices()` 메서드만 수정

## 3. 주식매매 영향 평가

### 🟢 매수 판단 — 영향 없음

| 요소 | 현재 | 변경 후 | 영향 |
|------|------|---------|------|
| last bar close | REST 10분 주기 | WS **실시간** (sub-sec) | ✅ **정확도 향상** |
| last bar high/low | REST 10분 주기 | WS **실시간 갱신** | ✅ **정확도 향상** |
| day volume | REST 10분 주기 | WS **실시간 갱신** | ✅ **정확도 향상** |
| ML 예측 입력 | daily bars (최신 포함) | 동일 (WS가 최신 유지) | 🟢 동일 |
| 스크리너 점수 | 최신 close 기준 | 동일 (WS가 최신 유지) | 🟢 동일 |

**핵심: 변경 후에도 일봉 데이터는 동일하게 유지됨. WS가 더 자주(last bar) 갱신하므로 오히려 정확도 향상.**

### 🟢 매도/청산 판단 — 영향 없음

청산은 보유종목 현재가(WS 체결가) 기반 — refresh_prices와 무관.

### 🟢 레짐 평가 — 영향 없음

레짐 = KOSPI 20일치 일봉. KOSPI 지수(`0001`)는 WS 구독 대상이 아니므로, 레짐 평가용 REST 호출은 유지됨.

## 4. 리스크 평가

| 리스크 | 심각도 | 확률 | 대응 |
|--------|--------|------|------|
| WS 연결 끊김 → REST 생략으로 데이터 누락 | **높음** | 30% | `ws_active` 플래그로 WS 연결 상태 실시간 감지. WS 끊기면 자동 REST fallback |
| 장 초반 WS 미구독 상태에서 REST 생략 | 중간 | 10% | 첫 refresh는 WS 구독 후 실행. WS 안정화 전까지 REST 호출 유지 |
| 장 마감 후 WS 데이터 outdated | 중간 | 20% | `is_market_hours()` 체크 추가 — 장외 시 WS 무시하고 REST 전량 호출 |
| WS 데이터 지연(체결가 늦게 도착) | 낮음 | 5% | WS on_price 호출 주기 0.1~0.5초. 10분 REST 간격 대비 충분히 빠름 |
| WS volume = 누적량 vs REST volume = 일봉 | 낮음 | 100% | WS volume은 당일 누적. REST도 당일 volume. 동일하므로 문제 없음 |

### 🛡 방어 로직 (3중 안전장치)

```
Layer 1: WS connected? — 연결 안 됐으면 REST 전량 호출
Layer 2: prices[-1].date == today? — 오늘이 아니면 REST 호출
Layer 3: code in WS subscribed? — 구독 안 했으면 REST 호출
```

3개 조건이 모두 만족될 때만 REST 생략. **어느 하나라도 불확실하면 REST 호출 유지.**
