## 2. 우선순위별 상세

### 🔴 P0 — 즉시 수정 (현재 운용에 직접 영향)

#### P0-1: 매수 후 `current_positions` 미갱신 (High #6)

**파일**: `live/orchestrator.py`  
**증상**: 매수 체결 후 `held_codes.add(pred.code)`만 수행. `current_positions`와 `held_quantities`는 다음 사이클까지 갱신 안 됨.  
**운영 영향**: 같은 사이클 내에서 섹터 집중도 검사, 최대 보유 종목 수 검사, 상관관계 검사가 **이전 데이터로 동작**.  
  - 최대 보유 종목(max_holdings=10) 초과 가능
  - 동일 업종 N개 초과 진입 가능
  - 상관관계 미검증으로 리스크 증가  
**수정**: 체결 후 `current_positions`와 `held_quantities`를 즉시 갱신하거나, 최소한 `held_quantities` 업데이트.  
**리스크**: 낮음 (포지션 정보 추가만 함, 기존 로직 변경 없음)

#### P0-2: `_refine_exit` 분봉 캐시 미사용 (High #4)

**파일**: `live/orchestrator.py`  
**증상**: `_refine_exit()`이 `self._minute_cache`를 거치지 않고 직접 KIS 분봉 조회 API 호출. `_refine_entry()`는 캐시 우선이라 같은 사이클 내 중복 API 호출 발생.  
**운영 영향**: KIS Rate Limit(초당 20회) 소모 가속화. 한계 상황에서 매매 지연 또는 실패 가능.  
**수정**: `_refine_exit()`에서도 `_refine_entry()`와 동일하게 `self._minute_cache` 우선 조회 후 캐시 미스 시에만 API 호출.  
**리스크**: 낮음 (캐시 로직 일치만 시킴)

#### P0-3: Idempotency 키 `(code, side)` 단순 (Medium #10)

**파일**: `live/executor.py`  
**증상**: `_submitted_orders[(code, side)]`로 중복 검사. 같은 종목 같은 방향 두 번째 매수는 중복으로 간주되어 차단.  
**운영 영향**: 동일 종목 추가 매수(분할 매수/평단 낮추기)가 불가능. 실제 거래 전략 제약.  
**수정**: idempotency 키에 `(code, side, timestamp)` 또는 `(code, side, quantity)` 포함. 또는 사이클 단위로 키 초기화.  
**리스크**: 중 (idempotency 보호 약화). 완전 제거보다 타임스탬프 결합 권장.

### 🟠 P1 — 오늘/내일 (시스템 안정성)

#### P1-1: `main.py` `data_provider` NameError (Critical #3)

**파일**: `main.py`  
**증상**: `finally` 블록에서 `data_provider.close()` 호출 시 `data_provider`가 `trading_loop()` 지역 변수여서 루프 진입 전 예외면 NameError.  
**운영 영향**: 시스템 종료 시 크래시 → Telegram 알림 누락, DB 연결 미종료 가능.  
**수정**: `data_provider`를 `run_live` 스코프 변수로 승격 + None 체크.  
**리스크**: 낮음 (변수 스코프 변경, None 체크 추가)

#### P1-2: `daily_loss_pct=0`과 `None` 구분 불가 (Medium #9)

**파일**: `live/risk.py`  
**증상**: `loss_pct = account.daily_loss_pct or account.total_profit_loss_pct` → daily_loss_pct=0(손실 0)도 None과 동일 처리.  
**운영 영향**: 손실이 전혀 없는 날에도 total_profit_loss_pct로 대체되어 잘못된 리스크 중단 가능.  
**수정**: `is not None` 체크로 변경.  
**리스크**: 없음 (조건식 변경만)

#### P1-3: 스크리너 config 새 인스턴스 생성 (High #5)

**파일**: `strategy/screener.py`  
**증상**: `_evaluate_single_stock()` 내부에서 `cfg = _StrategyConfig()`로 새 인스턴스 생성. 전달받은 config 무시.  
**운영 영향**: 대시보드에서 KOSPI 가산점, 레짐별 min_score 등 동적 변경이 스크리너에 반영되지 않음.  
**수정**: 전달받은 `config` 파라미터 사용하도록 1행 변경.  
**리스크**: 없음 (파라미터 사용으로 변경)
