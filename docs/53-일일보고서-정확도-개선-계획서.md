# 일일 보고서 정확도 개선 + Status 메시지 조건 수정 계획서

**작성일**: 2026-07-06  
**관련 이슈**: 
- P&L 계좌평가손익→체결기반 변경 완료 (`2c8203c`)
- 체결가 기록이 제출 시점 현재가여서 실제 손익과 불일치 (본 건)
- 장 마감 후 Status 메시지 계속 발송

---

## 1. 문제 정의

### 1.1 체결가 기록 불일치

**증상**: 일일 보고서 손익이 +500원으로 표시되나, 실제 체결 기준 손익은 +2,750원

**원인**: `live/executor.py`의 `execute_buy()` / `execute_sell()`에서 주문 제출 직후 `broker.get_price(code)`로 현재가를 조회하여 `fill_price`로 사용. 실제 KIS 체결가는 이와 다를 수 있음.

```
주문 제출 → KIS 접수(ODNO) → 현재가 조회(get_price) → save_trade(price=현재가)
                                                              ↑
                                                    실제 체결가 아님!
```

KIS 시장가 주문은 제출 즉시 호가창에서 체결되지만, NGSAT은 체결 완료를 확인하지 않고 현재가를 조회해 기록함. 변동성 장(ATR 5.3%)에서 시장가-체결가 괴리 발생.

### 1.2 장 마감 후 Status 메시지

**증상**: 장 종료(15:30) 후에도 10분마다 Status 메시지가 Telegram으로 발송됨

**원인**: `main.py`의 `_send_status_if_due()`가 `is_market_hours()` 체크 없이 시간 조건(600초 경과)만 확인

## 2. 수정 방안

### 2.1 A안: 체결 후 실제 체결가 조회 (executor.py)

**변경 대상**: `live/executor.py` — `execute_buy()`와 `execute_sell()`

**변경 내용**:

```python
# execute_buy() / execute_sell() 내부
order_id = await self._submit_with_retry(...)

# BE-9 (기존): 시장가 주문 현재가 조회
if adapted_price is None or adapted_price == 0:
    try:
        price_data = await self._broker.get_price(code)
        fill_price = price_data.close
    except Exception:
        fill_price = 0
else:
    fill_price = adapted_price

# --- 新增: 실제 체결가 조회 ---
try:
    await asyncio.sleep(0.5)  # KIS 체결 처리 대기
    status = await self._broker.get_order_status(order_id)
    if status in (OrderStatus.FILLED, OrderStatus.PARTIALLY_FILLED):
        # 체결됨 → 실제 체결가로 갱신
        fill_price = await self._broker.get_fill_price(order_id)
except Exception:
    pass  # 실패해도 기존 fill_price 유지
```

**필요한 것**: `BrokerAdapter`에 `get_fill_price(order_id)` 메서드 추가 — KIS 체결가 조회 API (CTSC9115R) 활용

#### 2.1.1 broker adapter에 get_fill_price 추가

KIS `CTSC9115R` (주문체결내역조회) 또는 기존 `inquire_order` 응답에서 체결가 추출.

```python
# adapter.py
async def get_fill_price(self, order_id: str) -> float:
    """KIS 주문체결내역 조회 → 실제 체결가 반환."""
    resp = await self._http.get("inquire_order", params={...})
    if resp.success:
        output = resp.raw.get("output", {}) or {}
        return float(output.get("ftrd_ccld_prpr", 0) or output.get("ccld_prpr", 0))
    raise BrokerError(f"체결가 조회 실패: {order_id}")
```

#### 2.1.2 ExecutionResult에 fill_price 필드 추가

```python
@dataclass
class ExecutionResult:
    ...
    fill_price: float = 0.0  # 실제 체결가 (0이면 price와 동일)
```

#### 2.1.3 save_trade()에서 fill_price 사용

orchestrator에서 `save_trade()` 호출 시 `fill_price`가 있으면 우선 사용.

## 3. 리스크 분석

### 3.1 A안 리스크

| # | 리스크 | 영향 | 확률 | 대응 |
|:-:|:-------|:----:|:----:|:-----|
| R1 | `get_order_status()` KIS API 추가 호출 → Rate Limit 소진 | 다른 API 호출 지연 | 중 | 0.5초 sleep 후 1회만 호출. 실패 시 기존 price 유지 |
| R2 | 주문 접수 직후 체결 미완료 상태 | FILLED 아닌 상태 반환 → 체결가 미확보 | 중 | `PARTIALLY_FILLED`도 체결가 있으면 사용. 미체결 시 기존 price 유지 |
| R3 | `asyncio.sleep(0.5)`로 사이클 0.5초 지연 | 전체 사이클 시간 증가 | 낮음 | 0.5초는 tick_interval=10s 대비 5% |
| R4 | `get_fill_price()` 구현 실수로 잘못된 가격 저장 | 손익 왜곡 | 낮음 | price와 fill_price 모두 보관, fallback 체계로 안전 |
| R5 | `ExecutionResult.fill_price` 추가로 기존 코드 영향 | 없음 (기본값 0.0, 하위 호환) | 없음 | dataclass 필드 추가만으로 기존 코드 변경 불필요 |

| # | 리스크 | 영향 | 확률 | 대응 |
|:-:|:-------|:----:|:----:|:-----|
