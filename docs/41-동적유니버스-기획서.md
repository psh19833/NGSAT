## 1. 배경

### 1.1 현재 문제

현재 NGSAT의 유니버스(관심종목) 40종목은 **코드에 하드코딩**되어 있어 절대 변하지 않습니다.

```
real_data_provider.py:
  KOSPI_CODES = ["005930", "000660", ...]  # 28종목 (고정)
  KOSDAQ_CODES = ["247540", "196170", ...]  # 12종목 (고정)
```

| 문제점 | 영향 |
|:------|:------|
| 추세 나쁜 종목이 계속 남음 | ML 평가만 낭비 |
| 새로 뜨는 종목을 못 담음 | 매매 기회 손실 |
| 시장 변화 미반영 | 3월에 정한 종목을 12월까지 사용 |

### 1.2 제약 조건

| 제약 | 내용 |
|:----|:------|
| **WebSocket 구독** | 최대 **41종목** (KIS 정책) |
| **REST API** | 초당 20회 (EGW00201) |
| **분봉 데이터** | WebSocket MinuteBarBuilder로 API 0회 수집 완료 |

## 2. 접근법: 장 시작 후 10분 관망 후 선정

### 2.1 핵심 아이디어

```
09:00 장 시작
09:00~09:10 데이터 수집만 (매매 없음)
   ─ WebSocket 40종목(기존 유니버스) 체결가 수집
   ─ KIS 거래량순위 API 1회 호출 (실시간 거래량 기반)
   ─ 분봉 데이터 10분치 누적

09:10 유니버스 선정
   ─ 거래량순위 상위 + 일봉 스크리너 점수 → 40종목 최종 선정
   ─ 기존 보유 포지션은 무조건 유지 (청산 방지)
   ─ 신규 편입 종목: WebSocket 구독 추가
   ─ 제외 종목: WebSocket 구독 해제

09:10~15:30 본격 매매
   ─ WebSocket 40종목 실시간 체결가 → MinuteBarBuilder
   ─ 분봉 스크리너 + ML 예측
```

### 2.2 거래량순위 API

KIS `거래량순위` API (1회 REST 호출):

| 파라미터 | 값 | 의미 |
|:---------|:---|:-----|
| `FID_COND_MRKT_DIV_CODE` | J | KRX 시장 |
| `FID_COND_SCR_DIV_CODE` | 20171 | 거래량순위 화면 |
| `FID_INPUT_ISCD` | 0000 | 전체시장 |
| `FID_DIV_CLS_CODE` | 0 | 전체 |
| `FID_BLNG_CLS_CODE` | 0 | 전체 |
| `FID_TRGT_CLS_CODE` | 000000000 | 전체 대상 |
| `FID_TRGT_EXLS_CLS_CODE` | 0 | 제외 없음 |
| `FID_INPUT_PRICE_1` | "" | 가격 하한 없음 |
| `FID_INPUT_PRICE_2` | "" | 가격 상한 없음 |
| `FID_VOL_CNT` | "" | 거래량 조건 없음 |
| `FID_INPUT_DATE_1` | (오늘 날짜) | |

**응답:** 거래량 상위 N개 종목코드 + 종목명 + 거래량 리스트

## 3. 상세 타임라인

```
08:50 서버 기동
  ├─ 기존 40종목 WebSocket 구독
  ├─ 일봉 데이터 로드 (360일)
  └─ 스크리너 + ML 모델 준비

09:00 장 시작
  ├─ WebSocket 체결가 수신 시작 (기존 40종목)
  └─ MinuteBarBuilder.feed() 시작 (매매 없음)

09:00~09:08 관망 (매매 보류)
  ├─ 분봉 8개 누적 (REST API 0회)
  └─ 거래량 데이터 수집

09:08 거래량순위 API 호출 (REST 1회)
  └─ 상위 60종목 응답 수신

09:09 유니버스 재선정
  ├─ 거래량순위 60종목 중 일봉 스크리너 점수 상위 35종목
  ├─ 기존 보유 포지션 종목 5종목 (무조건 유지)
  └─ 최종 40종목 확정

09:09:30 WebSocket 구독 교체
  ├─ 제외 대상: unsubscribe()
  ├─ 신규 편입: subscribe()
  └─ 교체 완료 후 MinuteBarBuilder 리셋

09:10 매매 시작
  ├─ 분봉 스크리너 (40종목)
  ├─ ML 예측
  └─ 매매 판단 실행
```

### 09:10 이전 매매 차단

`orchestrator.py`에서 `is_market_hours()`와 별도로 **진입 가능 시간** 체크 필요:

```python
TRADING_START_TIME = datetime.now(KST).replace(hour=9, minute=10, second=0)

async def run_cycle(...):
    if datetime.now(KST) < TRADING_START_TIME:
        # 데이터만 수집, 매매는 보류
        return
```

## 4. WebSocket 구독 교체 로직

### 4.1 구독 제한 41개 관리

| 구분 | 수 | 설명 |
|:----|:--:|:-----|
| KOSPI 지수 | 1 | 실시간 지수 확인용 |
| 거래량순위 상위 | 35 | 시장 활동성 기준 |
| 보유 포지션 | 5 | 기존 보유 종목 유지 |
| **합계** | **41** | **제한 내 최대 활용** |

### 4.2 교체 주기

| 주기 | 작업 | REST 호출 |
|:----|:-----|:--------:|
| **장 시작 +10분** (1일 1회) | 거래량순위 조회 + 40종목 교체 | 1회 |
| **장중** | 추가 교체 없음 (40종목 고정) | 0회 |

WebSocket 구독/해제는 REST 호출이 아니라 WebSocket 메시지이므로 Rate Limit에 영향 없음.
