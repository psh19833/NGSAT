# NGSAT (New Generation Stock Auto Trader) 기획서

> **문서 버전**: 1.0  
> **작성일**: 2026-06-25  
> **작성 방법**: Ouroboros 소크라테스식 인터뷰 (Wonder → Ontology → Design → Evaluation)  
> **면담자**: 대표님 (SAT3 운영자)

---

## 1. 프로젝트 개요

### 1.1 프로젝트명
NGSAT — New Generation Stock Auto Trader

### 1.2 목적
SAT3(StockAutoTrader-V3)의 구조적 한계와 전략적 한계를 모두 극복하는, 완전히 새로운 한국 주식 자동매매 시스템을 개발한다.

### 1.3 배경
SAT3 운영 경험에서 다음 문제가 확인되었다:
- **구조적 한계**: 기획 없이 개발이 진행되어 유지보수가 어려웠음
- **모듈화 부재**: 단일 모듈에 너무 많은 기능이 적재되어 코드가 얽히고, 기능 추가 시 연쇄 문제 발생
- **전략적 한계**: 매매 전략, 정책, 진입 타이밍 모두 성과가 부족했음
- **코드 오염**: 테스트 코드와 실거래 코드가 공유되어 테스트 잔여물이 실거래에 섞이는 문제 발생

### 1.4 성공 기준 (우선순위 순)
1. **수익률** (최우선) — 시장에서 실제 수익을 창출해야 함
2. **유지보수 편의성** — 모듈화된 구조로 기능 추가/수정이 독립적으로 가능해야 함
3. **안정성** — 오류가 적고 장시간 가동에도 안정적으로 운영되어야 함

## 2. 핵심 원칙

### 원칙 1: 모든 매매 판단에는 무조건 근거가 있어야 한다
- 매수, 매도, 홀드, 손절 등 **모든 거래 결정에 판단 근거가 명시**되어야 함
- 동적 손절 시에도 근거 필수 — 근거 없는 손절 연장 금지
- 근거는 데이터베이스에 기록되고 대시보드와 텔레그램 알림에 포함되어야 함
- 이 원칙은 NGSAT 전체 설계의 기둥임

### 원칙 2: 완전한 모듈화
- 한 모듈 = 한 책임 (Single Responsibility Principle)
- 모듈 간 결합도 최소화, 기능 추가 시 다른 모듈에 영향이 가지 않도록 설계

### 원칙 3: 실거래와 백테스트의 완전 분리
- 같은 저장소 내에서 별도 패키지로 물리적 분리
- live/ 패키지와 backtest/ 패키지는 서로 import할 수 없음
- 공통 코드(core, data, strategy, ml)만 공유
- SAT3의 "테스트 잔여물 문제"를 구조적으로 원천 차단

## 3. 매매 전략

### 3.1 전략 개요
SAT3의 전략을 계승하지 않고, 완전히 새로운 리서치를 통해 재설계한다.

### 3.2 3단계 파이프라인

```
1단계: 시장 레짐 평가
  ↓
2단계: 기술적 지표 + 차트 패턴으로 종목 선별
  ↓
3단계: 머신러닝으로 매매 결정
```

**1단계 — 시장 레짐 평가**
- 시장 상황을 강세/약세/횡보로 평가
- 세부 평가 방식은 리서치 단계에서 결정

**2단계 — 종목 선별**
- 기술적 지표와 차트 패턴을 조합하여 후보 종목 선별
- 세부 지표와 패턴은 리서치 단계에서 결정

**3단계 — 머신러닝 매매 결정**
- 가격 상승 확률 예측 → 진입 타이밍 판단
- 청산 타이밍 예측 → 매도 시점 판단
- ML 모델 구조, 학습 데이터, 피처는 리서치 단계에서 결정

### 3.3 전략 검증
- 상기 파이프라인의 유효성은 리서치 단계에서 검증
- 검증된 결과가 백테스트 모듈을 통해 성과 확인 후 실거래에 적용

## 4. 리스크 관리

### 4.1 일일 총손실 제한
- 하루 총손실이 전체 계좌 금액의 **-5%** 도달 시 **자동으로 매매 중단**

### 4.2 종목별 손절
- **기본 손절선: -3%**
- 프로그램이 상황을 판단하여 **추가 -2%까지 여유 부여 가능 (최대 -5%)**
- 단, 동적 손절 연장 시 **반드시 근거가 있어야 함** (원칙 1)

### 4.3 손실 안전장치 설계
- 손절 라인은 고정이 아닌 프로그램이 상황에 따라 동적으로 조절
- 모든 손절 결정은 근거와 함께 기록됨

---

## 5. 운영 모델

### 5.1 자동매매 제어
- **시작 / 멈춤 / 종료**: 사람이 버튼으로 제어
- **매매 자체**: 프로그램이 완전 자동으로 수행 (종목 선별 → 매수 → 매도)
- 핵심: "매매는 자동, 운영은 사람"

### 5.2 개입 기능
- **강제 매도 버튼**: 보유 종목을 즉시 매도
- **강제 홀드 버튼**: 특정 종목의 자동 매도를 일시 중단

### 5.3 매매 범위
- **종목 수 제한**: 없음 — 프로그램이 알아서 판단
- **매매 빈도 제한**: 없음 — 기회가 보이면 언제든 매수, 조건이 맞으면 언제든 매도
- **대상 시장**: 코스피 + 코스닥 (코스피 비중을 더 높게 설정)

## 6. 기술 아키텍처

### 6.1 프로젝트 구조

```
ngsat/
├── core/           공통: 데이터 모델, 유틸리티, 설정
├── data/           공통: 데이터 수집, 증권사 API 어댑터
├── strategy/       공통: 레짐 평가, 기술 지표, 차트 패턴
├── ml/             공통: 모델 학습, 추론, 피처 엔지니어링
├── live/           실거래 전용: 주문 실행, 리스크 관리, 운영 제어
├── backtest/       백테스트 전용: 과거 데이터 실행, 성과 분석
├── dashboard/      대시보드: 웹 UI
├── telegram/       텔레그램: 알림 및 원격 제어
└── docs/           문서
```

- **live/ ↔ backtest/ 상호 import 금지**
- core, data, strategy, ml 패키지만 양쪽에서 공유

### 6.2 증권사 API 연동
- **KIS(한국투자증권) API로 시작**
- **어댑터 패턴** 적용 — 증권사 API를 추상화하는 인터페이스 레이어
- 추후 다른 증권사 API 추가 시 어댑터만 추가하면 됨
- **구독 서비스 확장 고려** — 다중 사용자, 다중 증권사 지원

### 6.3 데이터 저장
- **데이터베이스 사용** — 파일 기반 저장 방식은 사용하지 않음
- 매매 이력, 판단 근거, 계좌 정보, 시장 데이터를 체계적으로 저장
- 구독 서비스 시 다중 사용자 데이터 관리 가능

### 6.4 사용자 인터페이스
- **대시보드(웹 UI)**: 매매 현황, 계좌 정보, 판단 근거 시각화
- **텔레그램 알림**: 모바일에서 실시간 확인 및 원격 제어
- 두 채널 모두 필수 — 대표님이 컴퓨터 앞에만 있지 않기 때문

### 6.5 텔레그램 알림 항목
- **거래내역 알림**: 매수/매도 체결 시 즉시 통지 (종목명, 가격, 수량, 근거)
- **일일 보고**: 장 마감 후 당일 매매 요약
- **가동상태 이벤트**: 프로그램 시작/중단/오류 등 상태 변화 알림
- **추가 항목**: 리서치 단계에서 보완

## 7. 개발 우선순위

### 7.1 접근 방식: 안전한 기초부터
모듈화, 데이터베이스, API 연결 구조를 먼저 튼튼하게 구축하고, 그 위에 전략과 ML을 올린다.

### 7.2 예상 개발 단계

```
Phase 1: 프로젝트 기초 구조
  - 패키지 구조 설계 및 스캐폴드
  - 데이터베이스 스키마 설계
  - core/ 모듈 구현

Phase 2: 데이터 계층
  - KIS API 어댑터 구현
  - data/ 모듈 (데이터 수집, 저장)

Phase 3: 전략 리서치 및 구현
  - 시장 레짐 평가 리서치 및 구현
  - 기술 지표 / 차트 패턴 리서치 및 구현
  - strategy/ 모듈

Phase 4: 머신러닝 계층
  - ML 모델 리서치 및 설계
  - 피처 엔지니어링
  - ml/ 모듈 (학습, 추론)

Phase 5: 백테스트 시스템
  - backtest/ 모듈 구현
  - 전략 성과 검증

Phase 6: 실거래 시스템
  - live/ 모듈 (주문 실행, 리스크 관리)
  - 운영 제어 (시작/멈춤/종료, 강제매도, 강제홀드)

Phase 7: 사용자 인터페이스
  - 대시보드 구현
  - 텔레그램 봇 구현

Phase 8: 통합 및 운영 검증
  - 전체 시스템 통합
  - 실거래 운영 검증
```

## 9. SAT3 교훈 요약

NGSAT 설계에 직접 반영된 SAT3 운영 교훈:

| SAT3 문제 | NGSAT 해결 방안 |
|-----------|----------------|
| 기획 없이 개발 → 유지보수 악화 | 기획서 작성 후 개발 시작 |
| 단일 모듈 과다 적재 | 한 모듈 = 한 책임 원칙 |
| 코드 얽힘 → 연쇄 오류 | 모듈 간 결합도 최소화 |
| 테스트 코드와 실거래 코드 공유 | live/ ↔ backtest/ 물리적 분리 |
| 전략/정책/진입타이밍 부족 | 완전히 새로운 리서치 기반 재설계 |
| 테스트 잔여물 오염 | 패키지 구조로 import 차단 |
