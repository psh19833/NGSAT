## Part A: 즉시 수정 (P0, ZERO Risk)

### A-1. 레짐 가중치 env 기본값 drift 수정 (1줄, ⏱ 1분)

| 항목 | 내용 |
|------|------|
| **파일** | `core/config.py:257` |
| **버그** | BUG-01: `load_config()`에서 `os.getenv("NGSAT_REGIME_WEIGHT_VOLUME", "15.0")` — dataclass 기본값은 `10.0`인데 env 기본값이 `15.0`. 환경변수 미설정 시 가중치 합=105 |
| **영향** | 🟡 레짐 점수 5% 과대평가 → BULL 임계(65) 조기 진입 → 과열 시장에서 과잉 매수 위험 |
| **리스크** | 🟢 **ZERO** — 단순 기본값 문자열 수정. env 파일에 이미 값이 있으면 영향 없음 |
| **처리** | `"15.0"` → `"10.0"` (dataclass 기본값과 일치) |
| **롤백** | 원복 |
| **검증** | `load_config()` 호출 시 `regime_weight_volume=10.0` 확인 |

## Part B: 단기 개선 (P1, LOW Risk)

### B-1. 분봉 모델 부재 시 일봉 fallback (⏱ 30분)

| 항목 | 내용 |
|------|------|
| **파일** | `ml/inference.py:290-291, 359-360`, `live/orchestrator.py` |
| **버그** | BUG-03: `has_minute_model=False`일 때 `predict_minute_entry()`/`predict_minute_exit()`이 `RuntimeError` raise. orchestrator에서 분봉 모델 없이 `mode=short_term`이 선택되면 사이클 전체 중단 가능 |
| **영향** | 🟡 분봉 모델 미저장 상태에서 short_term 모드 진입 시 사이클 사망 |
| **리스크** | 🟢 **LOW** — 예외를 fallback으로 대체. 기존 호출자는 raise에 의존하지 않음 |
| **처리** | **경량안 (추천)**: `RuntimeError` raise 대신 자동으로 일봉 `predict_entry()`/`predict_exit()` 폴백. 로그 경고만 남기고 정상 진행 |
| **검증** | `has_minute_model=False` 상태에서 `predict_minute_entry()`가 일봉 predict로 fallback되는지 확인 |

### B-2. `forward_days`/`forward_minutes` 필드 분리 (⏱ 20분)

| 항목 | 내용 |
|------|------|
| **파일** | `strategy/mode_selector.py:42,127`, `core/types.py` (`ModeDecision`) |
| **버그** | BUG-06: `ModeDecision.forward_days` 필드에 단타 모드에서 `ml_short_forward_minutes`(60=분) 저장. 필드명은 일수지만 실제로는 분 단위 값 |
| **영향** | 🔵 대시보드에 `forward_days=60` 표시 → 사용자가 "60일 후 예측"으로 오해 가능 |
| **리스크** | 🟢 **LOW** — dataclass 필드 추가, 기존 필드는 deprecated 처리 |
| **처리** | `ModeDecision`에 `forward_minutes: int \| None = None` 필드 추가. `mode_selector.py`에서 단타일 때 `forward_minutes` 설정, 스윙일 때 `forward_days` 설정 |
| **검증** | 단타 모드에서 `forward_minutes=60`, `forward_days=3` 확인 |

### B-3. `held_quantities` dict로 수량 기반 다중 매수 방지 (⏱ 20분)

| 항목 | 내용 |
|------|------|
| **파일** | `live/orchestrator.py:308,475-479` |
| **버그** | BUG-07: `held_codes`가 `set[str]`로 종목코드만 추적. 부분 익절 후 잔여수량 있는 상태에서 동일 종목 재매수 가능 |
| **영향** | 🔵 중복 매수로 평단가 희석 → 손절 시 손실 확대 |
| **리스크** | 🟢 **LOW** — dict로 변경만으로 quantity 추적 가능. 단, `held_codes` 참조하는 다른 코드 영향 확인 필요 |
| **처리** | `held_codes: set[str]` 유지 (호환성) + `_held_quantities: dict[str, int]` 추가. 매수 시 += quantity, 청산 후 차감. 매수 조건에 `held_quantities.get(code, 0) == 0` 추가 |
| **검증** | 부분 익절 후 동일 종목 재매수 시도가 차단되는지 확인 |

## Part C: 중기 개선 (P2, LOW-MEDIUM Risk)

### C-1. 모델 메타데이터에 `data_source` 저장 + 합성 모델 차단 (⏱ 1시간)

| 항목 | 내용 |
|------|------|
| **파일** | `ml/training/trainer.py` (`PriceRiseModel`), `ml/training/persistence.py`, `main.py` |
| **버그** | BUG-04: 모델이 합성 데이터로 학습됐는지 추적 불가. `orchestrator` 단계의 합성 차단(prefix 기반)은 작동하나, 백테스트에서 학습된 모델 자체가 합성 데이터 기반일 위험 존재 |
| **영향** | 🟡 운영자가 "KIS 실데이터 모델"이라 믿고 합성 데이터 모델로 실거래할 위험 |
| **리스크** | 🔵 **MEDIUM** — 모델 저장/로드 포맷 변경. 기존 저장 모델과의 하위호환성 필요 |
| **처리** | 1. `PriceRiseModel.__init__()`에 `data_source: str = "kis"` 파라미터 추가<br>2. `build_training_dataset()` 호출 전에 data_source 결정 로직 추가<br>3. `to_dict()`/`from_dict()`에 `data_source` 직렬화 포함<br>4. `main.py` 모델 로드 시 `data_source == "synthetic"`이면 경고 로그 + 사용 거부 |
| **검증** | 합성 데이터로 학습된 모델 로드 시 거부 메시지 확인 |

### C-2. `refresh_counter` 매직넘버 → `asyncio.Event()` (⏱ 30분)

| 항목 | 내용 |
|------|------|
| **파일** | `main.py` (trading_loop 내부) |
| **버그** | BUG-08: `_refresh_counter == 35` 매직넘버. tick_interval 변경 시 의미 상실. "재학습 후 fresh 데이터 적용" 의도가 코드에 드러나지 않음 |
| **영향** | 🔵 tick_interval 변경 시 silent break. 유지보수성 저하 |
| **리스크** | 🔵 **MEDIUM** — 이벤트 기반 로직 변경. 기존 카운터 기반보다 복잡 |
| **처리** | `_refresh_needed = asyncio.Event()` 추가. 재학습 완료 시 `set()`. 메인 루프에서 `_refresh_counter % 300 == 0` 조건과 함께 `if _refresh_needed.is_set()` 병행 체크. 또는 **경량안**: 매직넘버를 명명 상수로 추출 (`REFRESH_AFTER_RETRAIN = 35`, `REFRESH_INTERVAL = 300`) |
| **검증** | 재학습 후 refresh가 발생하는지 확인 |

### C-3. `win_rate` 계산 로직 추가 (⏱ 30분)

| 항목 | 내용 |
|------|------|
| **파일** | `main.py:319`, `data/repository.py` (TradeRepository) |
| **버그** | BUG-12: 일일 보고서 `win_rate=0.0` 하드코딩 |
| **영향** | 🔵 텔레그램 일일 보고서 승률 항상 0% 표시 |
| **리스크** | 🟢 **LOW** — 계산 로직 추가만. 기존 코드 영향 없음 |
| **처리** | `TradeRepository`에 `get_win_rate(date_str)` 메서드 추가 (이익 매도 / 전체 매도 비율). `main.py`에서 호출하여 `win_rate` 전달 |
| **검증** | 일일 보고서에 승률이 0이 아닌 실제값으로 표시 확인 |
